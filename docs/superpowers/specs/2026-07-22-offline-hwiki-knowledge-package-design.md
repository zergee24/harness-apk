# Harness 离线 `.hwiki` 知识包与双史书一期设计

日期：2026-07-22

## 1. 决策状态

本设计定义 Harness 的通用离线知识包 `.hwiki`，以及《二十四史》《资治通鉴》两个首发知识库在桌面 Codex 与 Android 端的完整一期边界。

本设计确认以下产品关系：

- `.hagent` 与 `.hwiki` 是同级、全局安装、独立版本化的资产。
- 首页继续只有“会话”和“项目”，不新增第三种顶层模式。
- 会话与 Wiki 是多对多关系。会话保存允许使用的 Wiki 精确版本，系统在该范围内按轮自动检索。
- Wiki 不归项目所有。项目只提供可选的 Markdown、文件和 Git 工作空间。
- 项目会话可以同时使用多个 Wiki，并把研究结果通过现有审核流程写入项目 Markdown。
- `.hwiki` 底层兼容完整 Wiki，但一期不要求预生成完整 Wiki 页面。
- Android 不负责重建大规模索引。重型解析、归一化、语义加工、索引和打包都在 M4 Mac mini 上完成。

本设计不替代现有《基于会话身份的人格智能体 V2 设计》和《项目会话驱动 Markdown 记事本设计》，只增加知识资产及其与会话的关系。若人格会话同时挂载 Wiki，`.hagent` 继续负责“谁在说话”，`.hwiki` 负责“本轮可以查什么”。

## 2. 问题定义

把大部头原文直接切块后做全文检索，只能解决“原句中恰好出现相同词”的问题，不能稳定解决以下任务：

- 用户使用现代汉语提问，原文使用古代名称、异名或隐含表达。
- 用户只给出人物、官职、地名、年号或大致时期，需要先定位相关卷目。
- 一个问题横跨多卷、多书或两个知识库，需要控制上下文而不是塞入大量低价值片段。
- 用户要求比较《二十四史》与《资治通鉴》的记载，需要两边都有证据并能返回原文。
- 用户不聊天，只想像普通 Wiki 一样按目录浏览、搜索、打开来源和沿引用跳转。

完整 GraphRAG、手机本地 embedding 和全量生成 Wiki 页面可以提高部分能力，但会显著增加桌面构建成本、包体、移动端依赖、调试难度和事实漂移风险。第一期需要的是介于“纯切块 FTS”和“完整知识图谱”之间的检索型语义加工层。

## 3. 产品目标

### 3.1 一期目标

1. 用户可以在桌面 Codex 中把有权使用的本地资料构建为签名 `.hwiki`。
2. 用户可以通过 Android 系统文件选择器或“用 Harness 打开”安装 `.hwiki`。
3. 用户可以离线浏览知识库目录、搜索原文、打开引用位置。
4. 新会话自动获得用户预设的 Wiki 可用范围，不要求每轮手动挂载。
5. 系统在允许范围内自动判断本轮应检索哪些 Wiki，并只把高价值证据交给当前模型。
6. 对《二十四史》和《资治通鉴》的现代汉语、别名、时间和跨库比较问题，能稳定定位可核验原文。
7. 每条使用 Wiki 的回答都能查看真实引用，历史消息不因知识库升级而改变来源。
8. 项目会话可以基于 Wiki 证据提出 Markdown 变更，但 Wiki 本身保持只读且不归项目所有。

### 3.2 成功标准

一期不是以“生成了多少 Wiki 页面”衡量，而是以以下结果衡量：

- 找得到：现代汉语问题和古文原文之间有足够的召回能力。
- 选得准：有限上下文优先保留能区分人物、时间、事件和来源差异的证据。
- 说得清：回答可综合，但事实判断可追溯到原文。
- 查得回：用户能从回答一键回到知识库、卷目和具体原文。
- 用得顺：默认自动路由，只有范围变化才需要用户操作。
- 升得稳：版本升级不污染旧会话和旧引用。

## 4. 非目标

以下内容不进入一期：

- 不新增首页“知识库”或“Wiki”顶层模式。
- 不让 Android 对整库执行 OCR、embedding、实体抽取、摘要生成或 GraphRAG 建图。
- 不依赖桌面常驻服务、远程 RAG 服务或中心知识库服务。
- 不要求两个首发知识库生成完整百科页面、完整事实声明库或完整关系图谱。
- 不在手机上引入向量数据库或本地 embedding 模型。
- 不让模型访问会话允许范围之外的已安装 Wiki。
- 不允许在移动端编辑 Wiki 原文或把项目 Markdown 写回 `.hwiki`。
- 不把 Wiki 自动归属某个项目，也不新增 `Project <-> Wiki` 直接关系。
- 不内置或代替用户下载受版权、发行许可或平台规则限制的全文。
- 不通过混淆、拆词或加密规避对象存储和分发平台审核。首发以用户本地导入为完整可用路径。

## 5. 第一性原则

### 5.1 原文是事实根，语义资产只是路由器

原文 chunk 及其可读位置是回答和引用的事实根。摘要、术语、别名、提及、时间标注、跨库链接和未来生成页面只能帮助定位原文，不能成为无来源结论。

### 5.2 把构建复杂度留在桌面，把查询确定性留在手机

桌面可以耗时完成解析、语义加工和验证；Android 只打开已经验证的只读索引，执行有限、可解释、可回归的查询流程。

### 5.3 用户授权范围与系统自动选择分离

用户决定“哪些 Wiki 可以被本会话使用”，系统决定“本轮实际查哪些”。自动路由永远不能扩大用户授权范围。

### 5.4 会话保存边界，消息保存事实

会话保存当前允许的精确 Wiki 版本。每次回答另外保存本轮范围快照、实际检索的 Wiki 和最终引用。这样既能调整后续范围，也能复现历史消息。

### 5.5 格式通用，领域能力可扩展

`.hwiki` 核心不硬编码人物、事件、官职、地名或年号表。领域构建器通过开放的 `kind`、`facets`、`annotations` 和 `links` 表达历史领域能力，未来可以构建法律、技术文档、企业制度或个人资料 Wiki。

### 5.6 不用完整性制造假象

包必须声明实际能力。生成页面可以为零、部分或完整；UI 不得把部分页面伪装成完整 Wiki。没有生成页面时，目录、原文搜索、引用和会话检索仍完整可用。

## 6. 概念与信息架构

| 概念 | 产品含义 | 所有权 | 可写性 |
| --- | --- | --- | --- |
| `Agent` | “谁在说话”的身份资产 | 全局安装 | 只读 |
| `Wiki` | “可以查什么”的知识资产 | 全局安装 | 只读 |
| `Conversation` | 交互、授权范围和历史来源 | 可选归属项目 | 消息可写 |
| `Project` | 文件、Markdown、Git 和工作上下文 | 用户项目 | 可写 |
| `Markdown` | 会话驱动的项目成果 | 项目 | 审核后写入 |

首页保持：

```text
会话 | 项目
```

低频资产管理放在：

```text
设置
  |- 智能体包
  `- 知识库
```

知识库管理页负责安装、版本、默认可用范围、浏览、搜索和卸载，不承载第二套聊天入口。知识库详情可以“在新会话中使用”，本质上仍创建普通会话并预选该 Wiki。

## 7. 领域模型

```text
InstalledWikiVersion 0..N <----> 0..N Conversation
                                      |
                                      | projectId optional
                                      v
                                  0..1 Project

AssistantMessage 1 ---- 0..1 WikiRetrievalRun
WikiRetrievalRun 1 ---- 0..N WikiUsage
AssistantMessage 1 ---- 0..N WikiCitation
```

### 7.1 已安装版本

`installed_wiki_versions` 至少保存：

| 字段 | 含义 |
| --- | --- |
| `wikiId` | 发布者命名空间内稳定 ID |
| `version` | 单调递增整数版本 |
| `title` | 展示名称 |
| `schemaVersion` | `.hwiki` 协议版本 |
| `publisherKeyId` | 发布者公钥指纹 |
| `manifestJson` | 已验证 manifest |
| `packagePath` | 应用私有目录中的包路径 |
| `contentHash` | 整包内容摘要 |
| `installedAt` | 安装时间 |
| `enabledForNewConversations` | 是否进入新会话默认范围 |
| `state` | `READY / DISABLED / INVALID` |

唯一约束为 `(wikiId, version)`。同一 Wiki 的多个版本允许并存，但同一 `wikiId` 最多只有一个 `READY` 版本可以设置 `enabledForNewConversations=true`；切换默认版本必须在同一数据库事务中完成。

### 7.2 会话当前挂载

`conversation_wiki_mounts` 表达会话当前允许范围：

| 字段 | 含义 |
| --- | --- |
| `conversationId` | 会话 ID |
| `wikiId` | Wiki ID |
| `wikiVersion` | 精确版本，不使用浮动 latest |
| `enabled` | 当前是否允许自动路由 |
| `mountedAt` | 加入时间 |
| `updatedAt` | 最近变更时间 |

唯一约束为 `(conversationId, wikiId)`，并要求目标 `(wikiId, wikiVersion)` 已安装且可用。

新会话创建时，把所有 `enabledForNewConversations=true` 的最新启用版本复制为会话快照。以后安装新 Wiki、升级版本或改变全局默认，不自动改变已有会话。

### 7.3 每轮检索记录

`wiki_retrieval_runs` 与助手消息一对一，至少保存：

- 本轮允许范围快照。
- 用户显式的单轮范围覆盖，例如“只看《资治通鉴》”。
- 路由与排序算法版本。
- `NO_QUERY / NO_HIT / HIT / FAILED` 状态。
- 候选数、最终证据数和耗时，不保存用户 API 密钥。

`message_wiki_usages` 保存本轮实际查询过的每个 `wikiId + version`、命中数量和是否进入最终上下文。

`message_wiki_citations` 保存：

- 消息 ID、展示顺序和知识库精确版本。
- `documentId / sectionId / chunkId`。
- 可读书名、卷目路径和位置。
- 实际引用原文快照与其 SHA-256。
- 引用覆盖的回答位置。

引用快照只复制本轮实际使用的小段原文。即使包文件意外损坏，历史消息仍能展示当时引用；正常卸载仍受版本引用保护。

### 7.4 与 Agent 和 Project 的关系

- `agentId + agentVersion` 与 Wiki 挂载相互独立。
- Agent 自带语料继续服务于人物身份和立场，不自动显示为一个 `.hwiki`。
- 同一轮同时使用 Agent 证据和 Wiki 证据时，来源记录必须区分 `AGENT_CORPUS` 与 `WIKI`。
- 项目不保存 Wiki 默认值。项目内新会话仍按新会话默认范围创建，用户可在发送前调整。
- 项目 Markdown 继续只通过现有 diff 审核写入，不因检索命中而自动落盘。

## 8. `.hwiki` 包格式

### 8.1 容器结构

`.hwiki` 是不可执行的签名 ZIP 容器：

```text
manifest.json
content.sqlite
checksums.json
signature.json
attachments/        # 可选，首发包默认不包含
```

核心约束：

- 包内不允许脚本、动态库、模型插件、Provider 配置或任意可执行文件。
- `content.sqlite` 必须使用 Android 目标版本可读取的 SQLite 格式和 FTS4。
- manifest、SQLite、附件分别进入 checksums，签名覆盖 checksums 与 manifest。
- 导入前执行 ZIP 路径、条目数量、单条目大小、总解压大小和压缩比限制。
- 安装采用 staging、完整校验和目录原子切换，不暴露半安装版本。

### 8.2 Manifest

manifest 至少包含：

```json
{
  "type": "hwiki",
  "schemaVersion": 1,
  "wiki": {
    "id": "cn-history-zizhi-tongjian",
    "version": 1,
    "title": "资治通鉴",
    "language": ["zh-Hant", "zh-Hans"],
    "description": "按卷编排的《资治通鉴》离线知识库",
    "contentHash": "sha256:64-lowercase-hex"
  },
  "publisher": {
    "keyId": "ed25519:publisher-fingerprint",
    "name": "用户本地发布者"
  },
  "capabilities": {
    "sourceHierarchy": true,
    "sourceSearch": true,
    "hierarchicalSummaries": true,
    "termIndex": true,
    "temporalAnnotations": true,
    "crossWikiLinks": true,
    "generatedPages": "partial",
    "claimGraph": false,
    "vectorIndex": false,
    "sourceAttachments": false
  },
  "conceptNamespace": "cn-history-v1",
  "conceptRegistryHash": "sha256:64-lowercase-hex",
  "builder": {
    "name": "harness-wiki-builder",
    "version": "1",
    "profile": "history-v1"
  }
}
```

`generatedPages` 只允许 `none / partial / complete`。其余可选能力必须显式声明；运行端不能根据表是否碰巧存在来猜能力。

### 8.3 原文与原始载体

必须区分：

- **原文文本**：可浏览、可搜索、可引用的标准化数字文本，是 `.hwiki` 核心内容，不能为了减小包体而省略。
- **原始载体**：PDF、EPUB、扫描页或版式文件，只用于视觉核验，是可选附件，不参与一期检索。

一期《二十四史》和《资治通鉴》默认包包含完整原文文本，不包含扫描 PDF。若以后提供附件，必须保持引用 locator 到附件页码的映射，并通过 `sourceAttachments` 能力声明。

## 9. 通用 `content.sqlite` 模型

### 9.1 必需表

| 表 | 作用 |
| --- | --- |
| `documents` | 书、文档或数据源，保存标题、责任者、版次、语言、授权说明、来源哈希和开放 metadata |
| `sections` | 任意深度目录树，保存父子关系、顺序、路径和稳定 ID |
| `chunks` | 原文、检索归一化文本、顺序、locator 和内容哈希 |
| `summaries` | 文档、章节或其他目标的分层摘要及原文证据 |
| `terms` | 通用术语或概念，`kind` 为开放字符串 |
| `aliases` | 术语别名、异体、简称和检索归一化形式 |
| `mentions` | 术语在原文 chunk 中的提及、位置和置信度 |
| `annotations` | 时间、分类、属性等开放标注及证据 |
| `links` | 包内或跨包目标之间的通用关系及证据 |
| `evidence_refs` | 以 `ownerType + ownerId` 把摘要、术语、标注、链接或页面声明关联到原文 chunk |
| `source_locators` | 可读位置与原始文档位置映射 |
| `build_metadata` | 构建器、规则版本、统计和兼容性信息 |

核心表不出现 `person_table`、`event_table`、`dynasty_table` 等历史业务专用表。历史构建 profile 可以使用：

```text
terms.kind = person | place | office | era | event | work | other
annotations.kind = time_range | reign_year | dynasty | genre | confidence
links.kind = same_as | mentions | succeeds | located_in | related_to | parallel_account
```

这些值是 `history-v1` profile 的约定，不是 `.hwiki` 协议枚举。其他领域可以定义自己的值。

### 9.2 FTS 索引

一期至少预生成四条独立 FTS4 通道：

1. `chunks_original_fts`：原文精确词面搜索。
2. `chunks_normalized_fts`：简繁、异体、标点和常见格式归一化后的搜索。
3. `summaries_fts`：现代汉语分层摘要，用于先找章节。
4. `terms_aliases_fts`：规范名、别名和相关检索词。

中文检索字段由桌面端预生成稳定的规范词和 2/3-gram token，以空格分隔后写入 FTS 列；Android 不依赖系统 tokenizer 自动完成中文分词。查询端使用同一规则版本生成 token，规则版本不匹配时拒绝把该通道标记为可用。

原文和归一化文本并行保存。引用只能展示原文，不能展示归一化文本或 n-gram 字段冒充原文。

### 9.3 可选扩展表

完整 Wiki 能力通过可选扩展表提供：

- `pages`、`page_sections`、`page_links`：预生成 Wiki 页面与链接。
- `claims`、`claim_evidence`：结构化声明和逐条证据。
- `graph_nodes`、`graph_edges`、`communities`：通用关系图和社区摘要。
- `vectors`：未来的预计算向量或量化索引。

一期运行端必须能忽略未知可选表。新 schema 只能通过 manifest `schemaVersion` 升级，不能静默改变既有字段含义。

## 10. 双史书检索型语义加工

一期包粒度固定为两个 Wiki：

- 《二十四史》作为一个 `.hwiki`，内部以二十四部史书作为独立 documents，不拆成二十四个需要逐个挂载的包。
- 《资治通鉴》作为另一个 `.hwiki`，按卷和年代段组织。

这样用户只管理两个知识范围，检索层仍可以按具体史书和卷目做细粒度路由。

《二十四史》和《资治通鉴》一期必须完成以下七层加工，不能只交付目录、chunks 和单路 FTS：

### 10.1 双文本层

- 保存可展示的原文文本。
- 生成仅供检索的归一化文本，处理简繁、常见异体、空白、标点和数字格式。
- 归一化必须可重复，规则版本写入 `build_metadata`。
- 不改写、删减或覆盖原文。

### 10.2 稳定层级

- 《二十四史》至少保留“史书 -> 卷 -> 篇章/传记单元 -> chunk”。
- 《资治通鉴》至少保留“全书 -> 卷 -> 年代段 -> chunk”。
- 每个 ID 由来源身份、层级路径和内容哈希稳定生成，同版重建不得随机变化。

### 10.3 现代汉语分层摘要

- 每部书、每卷和可识别的篇章单元生成现代汉语摘要。
- 摘要用于路由，不直接作为最终引用。
- 每个摘要必须关联支撑它的原文 chunk；无证据摘要阻止发布。
- 摘要保留时间范围、主要对象和事件线索，不追求百科式文采。

### 10.4 术语、别名与提及

- 为人物、地名、官职、政权、年号、事件和作品生成规范 term。
- 别名、异名、字、号、谥号、庙号、古今地名和简繁形式进入 aliases。
- 每个 mention 必须回到具体 chunk，不允许只存在于模型生成列表中。
- 低置信度抽取可以保留，但不能参与高权重精确路由。

### 10.5 时间标注

时间信息同时保存：

- 原文时间表达，例如年号、纪年或相对时间。
- 可比较的标准时间范围。
- 转换规则与置信度。
- 支撑该转换的 chunk。

无法确定的时间保留原文表达并降低置信度，不强制猜测公历年份。

### 10.6 跨 Wiki 对齐

两个知识库使用同一 `conceptNamespace=cn-history-v1`。历史 profile 在联合构建工作区维护版本化 `concept-registry.jsonl`，同一人物、地点、政权或事件尽量共享稳定 concept key，各包通过外部 key 建立弱链接，不互相复制正文。两个 manifest 同时记录该 registry 快照的 SHA-256，配对验证时必须一致。

对齐是可修正的检索线索，不是不可争议的历史事实。每条非机械对齐保存证据、置信度和规则版本。升级单个 Wiki 时可以更新对齐，但不能改变另一个已安装版本的内容。

### 10.7 可追溯性

所有摘要、terms、aliases、mentions、annotations 和 links 必须直接或间接关联原文 chunk。构建验证器拒绝：

- 悬空证据 ID。
- 引用不存在的 section 或 document。
- 摘要有事实但无支撑 chunk。
- 跨库链接没有外部 concept key。
- locator 无法回到可读原文位置。

## 11. 桌面构建器与 Skill

仓库新增与 `agent-builder` 同级的：

```text
.agents/skills/wiki-builder/SKILL.md
scripts/wiki-builder.sh
```

第一期支持 Apple M4 Mac mini、macOS、`darwin/arm64`，由桌面 Codex 调用，不开发独立桌面 GUI 或常驻服务。

### 11.1 输入

支持用户有权使用的：

- TXT。
- Markdown。
- EPUB。
- 有可靠文本层的 PDF。

扫描 PDF 需要先完成 OCR 文本层加工；一期 Wiki 构建器不内置通用 OCR 流程。失败文件必须进入报告并阻止完整发布，不能静默跳过。

### 11.2 固定流水线

```text
prepare
  -> 解析、目录识别、原文归一化、稳定 ID、基础 FTS
enrich
  -> 分层摘要、术语、别名、提及、时间和跨库 concept key
validate
  -> 结构、来源、召回、引用、SQLite 兼容和安全检查
pack
  -> 生成 content.sqlite、manifest、checksums 和签名 .hwiki
```

确定性处理可断点续跑。语义加工按 document/section 拆分任务并缓存到 workspace；重跑单个失败单元不重新处理整库。

### 11.3 交互原则

Skill 只在真实阻塞时提问：

1. 首次一次确认 Wiki 名称、稳定 ID、版本、输入路径和使用权。
2. 目录、版次、时间或来源元数据有 unknown 时，按文件稳定排序汇总成一次问题。
3. validate 失败时只询问无法由本地证据解决的冲突。

正常 `prepare / enrich / validate / pack` 不逐步请求“是否继续”。默认推荐安装内容为完整原文文本加一期检索型语义层；原始 PDF/EPUB 附件默认不打入移动包，并在报告中解释这一选择。

### 11.4 构建报告

每次发布生成 `build-report.json` 和可读 `build-report.md`，至少报告：

- 输入文件数、字节数、提取成功率和原文字符数。
- document、section、chunk、summary、term、alias、mention、annotation 和 link 数量。
- 每个 FTS 通道大小和最终压缩/安装大小。
- 语义加工耗时、失败重试和模型/规则版本。
- 结构校验、引用校验和检索评测结果。
- 生成页面能力为 `none / partial / complete` 的真实状态。
- 原始载体是否包含，以及不包含时的原因。

不在构建前猜测 1 GB 原稿的最终体积。以最终可检索文本、索引和语义资产的实测字节为准。

## 12. Android 安装与管理

### 12.1 本地导入

支持：

- “设置 -> 知识库 -> 添加”调用 Storage Access Framework 选择一个或多个 `.hwiki`。
- 从 Finder、AirDrop、USB、网盘或聊天工具把文件交给系统后，通过“用 Harness 打开”进入同一安装预览。
- 不申请全盘存储权限，不扫描用户目录寻找包。

安装预览只显示名称、版本、发布者指纹、压缩大小、准确安装大小、能力和版本冲突。校验通过且空间足够时，一个“安装”操作完成事务。

### 12.2 存储与事务

安装目录固定为：

```text
files/wikis/<wikiId>/<version>/package.hwiki
files/wikis/<wikiId>/<version>/content.sqlite
```

安装前至少需要“准确安装大小 + staging 开销”的可用空间。先复制到私有 staging，验证签名、checksums、SQLite 结构和内容约束，再原子登记版本。失败时清理 staging，不影响已安装版本。

安装验证必须检查 `sqlite_master` 只包含协议允许的表、索引和视图，拒绝 trigger、未知虚拟表模块和异常 schema。运行时以只读、`query_only` 模式打开数据库，不执行包内 SQL、不启用扩展加载，也不允许 Compose 或 Provider 直接拼接查询语句。

### 12.3 管理页

每个 Wiki 紧凑展示：

- 标题、当前启用版本、安装大小和发布者。
- 是否默认用于新会话。
- 目录、搜索、生成页面等实际能力。
- “浏览”“版本”“移除”操作。

低频细节进入详情，不在列表堆叠介绍性文案。知识库详情提供“在新会话中使用”，创建后仍进入统一会话页。

### 12.4 更新与卸载

- 新版本侧向安装，校验成功后才可设为新会话默认版本。
- 已有会话继续挂载旧版本，不静默升级。
- 用户在会话中显式切换版本，只影响切换后的消息。
- 被任何会话挂载或历史消息引用的版本不能直接卸载。用户需先移除当前挂载；历史引用仍存在时继续保留版本。
- 包损坏或被系统异常移除时，不回退到其他版本。会话显示“原知识库版本不可用”，历史引用仍显示保存的原文快照。

远端 catalog 更新复用软件更新式的签名清单和 HTTPS 下载思路，但不作为双史书一期发布阻塞项。本地文件导入必须独立完整可用。后续远端源由用户显式添加，不在 APK 中写死敏感全文或固定政治类目录。

## 13. 会话范围与交互

### 13.1 默认范围

知识库管理页允许用户切换“用于新会话”。安装首个可用 Wiki 后默认开启；安装后续 Wiki 时沿用安装预览中的明确选择，不弹出第二次配置。

新会话创建时复制当时所有默认启用的精确版本。首发两个 Wiki 都启用时，输入区只显示一个紧凑入口：

```text
[知识库图标  自动 · 2]
```

不分别占用两个常驻按钮。

### 13.2 调整范围

点击入口打开多选面板，可：

- 开关已安装 Wiki。
- 查看当前固定版本。
- 在已安装版本间切换。
- 恢复全局默认范围。

发送首条消息后仍允许调整，因为研究范围会变化；变更只影响后续消息。历史消息通过每轮检索记录保持原范围和引用。

用户在问题中明确说“只看《资治通鉴》”“比较两边记载”时，系统把它解析为单轮范围/意图覆盖，不永久修改会话挂载。范围不明确时维持自动路由。

### 13.3 无 Wiki 与停用状态

- 会话没有挂载 Wiki 时走现有聊天链路，不显示空引用区。
- 挂载版本不可用时，该 Wiki 不参与检索，并在紧凑入口显示异常状态。
- 允许范围中只有一个 Wiki 时直接查询，不运行多库选择步骤。

## 14. 移动端自动路由与检索

一期不依赖模型 tool-calling。所有 Provider 共用应用内 `WikiQueryGateway`，其稳定逻辑接口为 `searchWiki` 和 `readSource`；会话编排器在模型回答前调用它们，保证不同模型都能获得相同的检索和引用能力。未来即使向支持工具调用的模型暴露同名工具，也必须复用该 Gateway 的授权、排序和引用边界。

### 14.1 查询流程

```text
用户问题
  -> 读取本轮允许的 Wiki 精确版本
  -> 查询归一化与显式范围/比较意图识别
  -> 各 Wiki 执行轻量 scout 检索
  -> 跨通道、跨 Wiki 排名融合
  -> 选择相关 Wiki 和候选 section
  -> 在候选 section 内检索原文 chunks
  -> 去重、补邻接、覆盖度选择
  -> 组装有原文证据的模型上下文
  -> 模型回答
  -> 引用校验、持久化和渲染
```

### 14.2 Scout 检索

对允许范围内每个 Wiki 并行查询：

- 现代汉语 summaries。
- terms 与 aliases。
- 时间 annotations。
- 原文 normalized FTS 的少量高分命中。

各通道的 SQLite/BM25 原始分数不可直接跨库比较。系统使用 Reciprocal Rank Fusion 合并名次，再结合精确别名、时间重叠、section 命中和用户显式书名生成路由分。

普通问题默认进入最高相关 Wiki；以下情况保留多个 Wiki：

- 用户明确要求比较或互证。
- 多库路由分接近且都超过最低门槛。
- 单库深检索没有足够证据，系统扩大到剩余允许 Wiki。

系统不能因为某个 Wiki 体积更大就天然提高其分数。

### 14.3 深检索

在候选 sections 内同时运行：

- 原文精确 FTS。
- 归一化原文 FTS。
- term/alias 到 mentions 的反向定位。
- 时间范围过滤或加权。
- 跨 Wiki concept key 的弱扩展。

多路结果继续用排名融合，不把低置信度生成关系当作硬过滤。引用 chunk 必须来自原文表。

### 14.4 信息密度控制

移动端优化目标不是最大召回片段数，而是单位上下文内的证据覆盖度。最终选择遵循：

- 默认最多 10 个原文 chunk，硬上限 12 个。
- 原文总量软目标为 6,000 个中文字符；模型上下文允许时硬上限为 12,000 字符。
- 同一 section 默认最多 3 个 chunk；只有连续原文或直接引语跨块时补相邻块。
- 比较问题至少为每个被比较 Wiki 保留 2 个有效 chunk；某一边无证据时明确显示缺口，不用另一边填充。
- 优先覆盖不同关键 term、时间、section 和来源，惩罚高度重复片段。
- summaries、terms 和 links 只用于选材，默认不进入最终模型上下文。

具体阈值由离线评测校准，但产品约束“有界、可解释、优先原文、比较时保留双方证据”不可改变。

### 14.5 无结果策略

第一次深检索不足时，系统自动：

1. 放宽别名和时间权重。
2. 从已选 sections 扩展到相邻层级。
3. 查询允许范围内尚未进入深检索的 Wiki。

仍无可靠证据时，模型收到明确的 `NO_EVIDENCE` 状态，只能说明当前已挂载资料未找到依据并提出一个必要追问。不得用模型常识伪造 Wiki 结论或引用。

## 15. 回答与引用协议

### 15.1 上下文安全

包内原文、摘要和元数据都按不可信资料处理，不能覆盖系统提示、切换身份、调用工具、修改设置或扩大知识库范围。一期只把最终选中的原文证据注入模型，其他语义资产留在本地参与检索。

模型只看到最终原文证据、可读 locator 和临时 citation token。内部 SQL、文件路径、BM25 分数和 chunk ID 不进入可见回答。

### 15.2 引用生成

- 模型使用临时 token 标注事实依据。
- 流式完成后，解析器验证 token 只引用本轮实际提供的 chunk。
- 无效 token 从可见正文移除，并把回答标记为“引用未完全验证”；不伪造来源。
- 直接引语必须是引用 chunk 中可匹配的连续原文，否则不按引语渲染。
- 多个相邻 token 可合并为一个可点击来源锚点。

正文使用低干扰的可点击引用锚点，不显示 `[资料8]`、内部 ID 或文件名堆叠。回答底部默认折叠显示：

```text
引用 6 · 二十四史 3 · 资治通鉴 3
```

展开后展示原文、书名、卷目路径、版本和位置。点击引用直接打开知识库阅读器并定位高亮原文。

### 15.3 生成内容边界

- 回答可以综合多个片段，不要求逐句复述。
- 历史事实、比较判断和直接引语必须有来源。
- “两书没有记载差异”只有在明确检索双方且证据足够时才能表达；未检索到不等于不存在。
- 生成页面、摘要或图关系不能单独支撑最终事实。
- 如果回答同时使用 Agent 语料和 Wiki，引用面板按来源类型分组，但正文保持统一表达。

## 16. 独立浏览体验

打开已安装 Wiki 后，即使不创建会话也可以使用：

### 16.1 目录

- 按 `documents -> sections` 展示任意深度目录。
- 面包屑保留当前位置。
- section 页面展示标题、可用摘要和连续原文。
- 上一节、下一节和返回上层使用图标操作，并提供无障碍标签。

### 16.2 搜索

- 单一搜索框同时查原文、归一化原文、摘要和别名。
- 结果优先显示可读路径和原文片段，命中别名时显示对应规范词。
- 搜索过滤只提供确有数据的开放 facets，例如文档、时期或 term kind，不把历史专用过滤写死到通用 UI。

### 16.3 生成页面

- `generatedPages=none` 时不显示“主题”入口。
- `partial` 时只展示实际生成的页面，并标明覆盖数量，不出现空白页面或自动补写。
- `complete` 时可以提供完整页面索引，但每段事实仍需跳转到原文证据。
- 页面缺失时始终回退到目录和原文检索，不阻断浏览。

### 16.4 引用回跳

从会话引用打开阅读器时：

1. 定位精确 `wikiId + version`。
2. 展开 document 和 section 路径。
3. 高亮实际引用文本。
4. 允许阅读前后文和从当前位置发起新会话。

## 17. 与项目 Markdown 的协作

Wiki 保持只读，项目 Markdown 是用户可持续加工的工作成果。项目会话中的数据流为：

```text
Wiki 检索原文
  -> 会话回答与引用
  -> 模型提出 Markdown 变更
  -> 用户查看 diff
  -> 应用到项目文件
```

规则：

- 只有 `projectId != null` 的会话进入现有 Markdown 自动规划链路。
- 写入内容引用 Wiki 时，提案使用普通 Markdown 脚注，至少包含书名、卷目、知识库版本和可读位置。
- Markdown 不写入只能由 Harness 解析的隐藏 front matter，也不复制完整 Wiki 页面。
- `conversation_markdown_links` 继续表达会话与 Markdown 的弱关联；不增加 Markdown 与 Wiki 的强关联表。
- Wiki 升级不自动重写项目 Markdown。需要更新依据时，由会话重新检索并提出新 diff。
- 普通会话挂载 Wiki 后仍不能静默创建项目或写入 Markdown。

## 18. 版本、兼容与恢复

### 18.1 Schema 兼容

- Android 声明可读取的 `schemaVersion` 范围。
- 高于支持范围的包在安装预览中拒绝，不能尝试部分安装。
- 同 schema 的新增可选 capability 必须向后兼容。
- SQLite 的必需表、列、索引和 FTS tokenizer 在安装验证阶段检查。

### 18.2 内容升级

- `wikiId` 标识同一知识库，`version` 标识不可变内容快照。
- 同一 `wikiId + version` 的不同 `contentHash` 视为冲突包，默认拒绝覆盖。
- 新会话使用用户为该 Wiki 当前启用的精确版本；系统不自行把“最高版本”解释为已授权升级。
- 旧会话升级需要用户显式选择目标版本；升级只作用于未来消息。
- 旧版本因引用保护保留，不由空间清理器自动删除。

### 18.3 恢复

- 安装中断后清理未登记 staging。
- 数据库打开或完整性检查失败时把版本标记为 `INVALID`，不参与路由。
- 已有可用版本不因新版本安装失败而停用。
- 检索异常只影响本轮 Wiki 增强；助手可以明确提示知识检索失败，但不能静默当作查无资料。

## 19. 安全、版权与隐私

- 桌面 Skill 首次构建必须确认用户有权使用输入资料，不替用户抓取受保护全文。
- 构建默认纯本地，不上传原文、workspace、签名私钥或构建产物。
- Android 本地浏览和检索不联网，也不依赖 Mac 在线。
- 最终回答仍使用用户当前配置的模型 Provider。若不是本地模型，只有最终选中的少量原文证据会随本轮请求发送给该 Provider；不会上传整库。
- 包签名只证明发布者和内容未被篡改，不代表史料观点或生成摘要绝对正确。
- 所有生成语义资产保留构建器版本和证据链，支持重新构建与审计。
- 远端分发若后续启用，采用用户可配置的签名 catalog；官方 APK、官方更新源与用户自有资料解耦。
- 架构不承诺绕过 OSS、CDN、应用商店或地区政策。敏感资料优先通过用户有权使用的本地文件导入和自托管源分发。

| `WikiCitationVerifier` | 验证、持久化和渲染引用 |
| `WikiBrowser` | 目录、搜索、页面和原文阅读 |

这些组件通过领域模型交互，不让 Compose 页面直接查询包内 SQLite，也不让 Provider 适配层各自实现一套 RAG。

## 21. 测试与质量门槛

### 21.1 桌面构建测试

必须覆盖：

1. 同一输入和规则版本重建时 document、section、chunk ID 稳定。
2. 原文字符覆盖率为 100%，提取失败文件阻止发布。
3. 每个 summary、mention、annotation 和 link 的证据可达。
4. 两个包的共享 concept key 格式一致，悬空外部链接可报告。
5. SQLite 能被 Android 目标环境以只读方式打开，四条 FTS 通道可查询。
6. checksums、签名、ZIP 限制和 schema 校验通过。
7. 中断恢复只重跑未完成单元，不污染已验证结果。

### 21.2 检索评测集

两个首发 Wiki 各自建立人工核验的评测集，并增加跨库集合。至少覆盖：

- 原文关键词。
- 现代汉语改写。
- 人物异名、谥号、庙号、官职和古今地名。
- 年号、朝代、相对时间和时间范围。
- 同名消歧。
- 单书多卷综合。
- 两书互证、差异和记载缺口。
- 明确无结果问题。

发布门槛：

- 单库 gold 证据 `Recall@20 >= 90%`，且每一类别不得低于 `85%`。
- 明确跨库比较问题中，最终 12 个 chunk 对双方 gold 来源的覆盖率 `>= 90%`。
- 直接引语与保存引用原文匹配率 `100%`。
- 引用 document、section、版本和 locator 有效率 `100%`。
- 无效或伪造 citation token 的可见通过率 `0%`。

若真实数据证明阈值不可达，必须修正构建或检索规则并重新评测，不能通过降低总门槛发布；类别样本和失败案例随构建报告保留。

### 21.3 Android 测试

必须覆盖：

1. 安全导入、空间不足、签名失败、冲突版本和中断回滚。
2. 新会话复制全局默认范围，已有会话不随全局设置变化。
3. 同一会话挂载两个 Wiki，单轮自然语言范围覆盖不永久改挂载。
4. 更新后旧消息仍打开旧版本引用，新消息按显式选择使用新版本。
5. 单库、双库、无结果和检索失败的回答路径。
6. 不支持 tool-calling 的 Provider 仍获得相同 Wiki 检索结果。
7. 引用 token 校验、引用折叠区和阅读器精确回跳。
8. `generatedPages=none / partial / complete` 的 UI 降级。
9. 项目会话能生成带可读脚注的 Markdown diff，普通会话不写项目。
10. 被会话或消息引用的版本不可卸载。

在 Android 目标测试环境中，两个首发 Wiki 同时启用时，热启动本地检索的 `P95` 目标为 1 秒内；冷打开单个包的首次检索目标为 2 秒内。性能统计只计算本地路由和检索，不包含外部模型响应时间。

## 22. 一期验收场景

一期只有同时通过以下场景才算完成：

1. 用户本地安装《二十四史》和《资治通鉴》两个签名 `.hwiki`，全程无需桌面服务在线。
2. 用户不聊天即可浏览两库目录、搜索原文并查看前后文。
3. 新会话默认显示“自动 · 2”，提问单书问题时只使用相关 Wiki。
4. 用户以现代汉语、人物异名或年号提问，回答能回到正确原文。
5. 用户要求比较两书同一事件，回答至少展示双方证据；某一方无可靠证据时明确说明检索缺口。
6. 用户说“这一轮只看《资治通鉴》”，本轮不访问《二十四史》，下一轮恢复会话原范围。
7. 用户升级《资治通鉴》后，旧消息仍打开旧版本原文，新消息只有在显式切换后使用新版本。
8. 用户从引用进入阅读器，能定位卷目、原文和前后文。
9. 项目会话可把比较结果生成为带脚注的 Markdown 变更，未点击应用时文件不改变。
10. 关闭网络后，安装后的目录、搜索、路由、检索和引用浏览仍可用；只有远程模型回答按现有 Provider 能力受网络影响。

## 23. 分期边界

### 一期必须交付

- 通用 schema v1 和签名 `.hwiki`。
- `wiki-builder` Skill 与桌面离线构建链路。
- 《二十四史》《资治通鉴》的检索型语义加工和评测。
- Android 本地安装、管理、浏览、搜索和版本保护。
- 会话多对多挂载、默认范围快照、自动路由和逐消息引用。
- 项目 Markdown 现有审核链路的 Wiki 脚注集成。

### 后续兼容但不阻塞一期

- 签名远端 catalog 与增量下载。
- 更多领域 profile。
- 部分或完整预生成 Wiki 页面。
- claims、图社区摘要和可视关系浏览。
- 桌面预计算向量与移动端量化向量检索。
- 原始 PDF/EPUB 附件包和页级视觉回跳。

## 24. 开源方案取舍

- [PageIndex](https://github.com/VectifyAI/PageIndex) 的长文档层级检索思路适合借鉴到“先章节、后原文”的路由流程。
- [Microsoft GraphRAG](https://github.com/microsoft/graphrag) 的实体关系、社区摘要和局部/全局查询适合作为桌面构建期的后续能力参考，不直接作为 Android 一期运行时。
- [OpenKB](https://github.com/VectifyAI/OpenKB) 可参考“知识导入后既可浏览又可问答”的产品形态，但 Harness 保持本地签名包和移动端只读边界。
- [RAGFlow](https://github.com/infiniflow/ragflow) 可参考解析可观察性和引用回溯体验，不引入其服务端运行形态。
- [LightRAG](https://github.com/HKUDS/LightRAG) 可作为未来图和向量混合检索的实验基线，不进入一期依赖。

最终路线是：吸收这些项目的层级、图谱、引用和检索经验，保持 Harness 的核心实现为“桌面预构建的通用签名 SQLite 包 + Android 有界离线检索”。

## 25. 实施拆分约束

本 Spec 是统一产品和协议边界，不应作为一个超大提交一次实现。设计获批后，实施计划按以下依赖顺序拆成可独立验收的工作包：

1. `.hwiki` schema、签名 fixture、桌面确定性索引与验证器。
2. Android 安装、版本管理、目录浏览、原文搜索和引用阅读器。
3. 会话多对多挂载、自动路由、深检索、回答引用和历史版本保护。
4. 项目 Markdown 脚注集成与双 Wiki 比较闭环。
5. 《二十四史》《资治通鉴》语义加工、联合 concept registry、评测和正式包构建。

每个工作包必须在进入下一个工作包前通过自身测试；最终再执行第 22 节的端到端验收。协议 fixture 和检索评测集先于业务 UI 固化，避免移动端与桌面构建器各自猜测格式。
