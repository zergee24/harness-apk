# 会话排队、引导与后台流式执行设计

## 背景

当前会话页面把发送任务创建在 `ChatScreen` 的 Compose 协程作用域中。它有三个直接问题：

1. 输入文字和待发送图片在整个流式回复完成后才清空，用户会误以为发送没有生效。
2. 页面离开组合、切换应用或锁屏时，协程可能被取消，OkHttp 流式请求随之中断。
3. 生成期间输入框的发送按钮被替换为终止按钮，用户无法自然地继续输入下一轮内容，也无法把一条已排队输入用于引导当前回复。

Codex App Server 的公开协议提供 `turn/steer`，可向执行中的 turn 追加用户输入。Harness APK 面向多家 OpenAI-compatible 服务商，现有 `chat/completions` 流式协议没有对应的注入能力，因此需要以“停止当前流后带引导续写”的方式实现同等用户体验。

参考资料：

- Codex App Server API overview，`turn/steer` 与 `turn/interrupt`：https://learn.chatgpt.com/docs/app-server#api-overview

## 目标

1. 用户点击发送后立即清空输入文字和待发送图片。
2. 每个会话支持 FIFO 排队；不同会话可同时进行流式请求。
3. 当前回复进行中，用户仍可输入和发送；新输入默认排入下一轮。
4. 用户可对任一排队项执行引导当前、编辑或删除。
5. 切换应用、锁屏或会话页面失去焦点时，流式请求继续执行，除非用户主动终止。
6. 思考过程默认以中文请求输出；折叠时显示最新非空一行和展开入口，展开后显示完整可见内容。
7. 已生成的部分回复在被引导打断后保留，并清晰标记为“已按引导结束”。

## 非目标

- 不接入 Codex App Server 或改造为单一厂商协议。
- 不伪造服务商未返回的思考内容，也不为已有外语思考内容额外发起翻译请求。
- 不在通知栏提供暂停或终止操作；终止仍在会话输入框内完成。
- 不保证进程被系统杀死后恢复同一条 HTTP 流；不能从断开的 SSE 连接继续读取。
- 不新增“推荐提示词卡片”。本设计的“引导”仅指用户在输入框输入的信息对当前回复的引导。

## 方案比较

### 方案 A：继续在 ChatScreen 直接执行

只把输入框提前清空，并在 UI 内维护列表队列。

优点：改动小。

缺点：页面释放或 Activity 重建仍会取消请求，后台运行、跨会话并行和可靠队列都无法满足。

### 方案 B：应用级前台服务与持久化任务队列

前台服务负责调度与执行；Room 保存任务和状态；Compose 页面仅观察状态并发出操作意图。

优点：满足失焦连续执行、每会话串行、跨会话并行、队列编辑和安全恢复。

缺点：增加 Room migration、前台服务通知和任务调度边界。

### 方案 C：迁移至支持中途 steer 的专有 API

直接使用具备 active-turn steer 能力的运行时协议。

优点：真正向同一个活动 turn 注入输入。

缺点：破坏多供应商兼容性，且不适用于当前的 OpenAI-compatible 提供商集合。

采用方案 B。对当前流的“引导”采用协议无关的等价语义：持久化已有输出，终止该流，然后立即以引导信息发起后续续写。

## 执行模型

### 会话级并发

- 同一 `conversationId` 同时至多一个活动流。
- 同一会话内按 `sequence` FIFO 执行任务。
- 不同会话允许并行流式执行，不设置全局串行锁。
- 单个任务的限流、超时或失败只影响自身；其他会话和后续任务继续。

### 前台服务

新增 `ChatExecutionService`：

- 当存在至少一个活动流或待执行队列项时启动为前台服务。
- 使用应用级 `SupervisorJob` 管理按会话分组的执行任务，UI 重组、路由切换和输入失焦不影响它。
- 通知显示“正在生成 N 个回复”，不提供操作按钮。
- 队列清空且不存在活动流时停止服务并移除通知。
- Manifest 声明相应的 foreground service 权限、`dataSync` 服务类型和通知 channel。

### 进程被杀死

- 活动 HTTP 流不能安全续接，恢复时将其标记为“已中断”，保留部分输出和“重新发送”入口。
- 未开始的任务保持 `QUEUED`，下次打开应用后由服务继续执行。
- 不自动重发已开始任务，避免模型工具调用、搜索或写文件类请求重复执行。

## 数据模型

新增 `chat_execution_entries` 表，并把数据库升级为下一版本。

| 字段 | 含义 |
|---|---|
| `id` | 队列任务 id |
| `conversationId` | 所属会话，建立索引 |
| `userMessageId` | 已持久化的用户消息 |
| `assistantMessageId` | 运行后创建的助手消息，可为空 |
| `sequence` | 会话内稳定 FIFO 顺序 |
| `type` | `NORMAL` 或 `STEER_CURRENT` |
| `status` | `QUEUED`、`RUNNING`、`SUCCEEDED`、`FAILED`、`CANCELLED`、`INTERRUPTED`、`STEERED` |
| `providerId` / `model` | 入队时冻结的模型选择 |
| `reasoningEffort` | 入队时冻结的推理档位 |
| `requestContextJson` | 项目、会话提示词、联网搜索等必要快照 |
| `targetAssistantMessageId` | 引导当前时被终止的助手消息 |
| `errorMessage` | 可读错误摘要 |
| `createdAt` / `updatedAt` | 生命周期时间 |

约束：

- `conversationId + sequence` 唯一。
- 每个会话最多一条 `RUNNING`。
- 任务状态转换和消息状态转换在同一 Room transaction 中完成。
- 历史任务保留用于渲染“等待处理”“已按引导结束”和失败重试；会话归档或删除时级联删除。

### 请求历史筛选

排队用户消息虽然已经显示在会话中，但在其真正开始执行前不得进入模型上下文。执行器构造请求时只读取：

- 已成功、失败或取消的历史消息。
- 当前正在执行的用户消息。
- 引导当前时已稳定的部分助手输出和被选中的引导消息。

同会话中其他 `QUEUED` 用户消息必须排除，避免模型提前看到未来轮次而破坏 FIFO 语义。

### 图片快照

入队前把相册或相机 URI 复制到应用私有目录，并让 `message_attachments` 指向受应用控制的文件。执行器从已持久化附件读取图片，而不是依赖输入框的临时 URI。

### 语言指令

在构造请求上下文时追加一条稳定的系统级偏好：可见回答和可见思考摘要默认使用中文。它不得覆盖用户明确指定的输出语言，也不得尝试读取或转换服务商未暴露的内部推理。

## 交互设计

### 输入框与尾部按钮

输入框不设置“排队 / 引导”模式开关。

| 条件 | 尾部按钮 | 行为 |
|---|---|---|
| 文字非空或已选择图片 | 发送 | 立即持久化为下一轮队列项，并清空输入 |
| 输入为空且当前会话有活动流 | 终止/暂停 | 终止当前会话的活动流 |
| 输入为空且无活动流 | 添加图片 | 打开拍照/相册入口 |

发送成功入队后，文字和图片预览在同一帧清空；若文件复制或入队失败，则保留草稿并显示错误。

### 队列项

排队用户消息直接出现在会话中，显示“等待处理”状态。点击队列项打开操作菜单：

- **引导当前**：将该项提升为当前会话下一个执行项；若有活动流，先请求终止并把已有助手输出标记为“已按引导结束”，再优先执行该项。其他队列项相对顺序不变。
- **编辑**：把该项内容与图片放回输入框，移除原任务；重新发送后按新的内容重新入队。
- **删除**：移除未执行任务和其关联的用户消息附件。

运行中的任务不可编辑或删除；用户可通过输入框空态的终止按钮停止它。

### 引导当前回复

`STEER_CURRENT` 不尝试向既有 SSE 连接写入数据：

1. 当前助手消息停止接收 delta，已落库的 text/reasoning parts 保持稳定。
2. 当前助手消息状态改为取消，并以执行状态展示“已按引导结束”。
3. 被选中的用户消息以优先级最高的下一个任务执行，模型请求带入已生成回复和该条引导内容。
4. 新助手消息紧跟引导消息产生；原队列其余项目继续 FIFO。

### 思考过程

- 默认折叠。
- 折叠态显示“思考过程”、最新非空一行摘要和展开/收起图标；摘要 `maxLines = 1` 并使用 ellipsis。
- 展开态展示完整已持久化 reasoning 内容。
- 流式期间摘要随最新 reasoning delta 更新，但不导致整个消息气泡重组。

## 错误与边界

- 任务入队前复制图片失败：不清空草稿，不创建队列项。
- 模型不支持图片：队列任务失败并显示可读错误，用户消息与图片仍保留。
- 当前任务失败：标记为失败并提供重试；同会话下一个任务继续。
- 引导与终止竞争：以同一会话的数据库事务串行化，避免同一任务被执行两次。
- 返回会话或服务重连后，UI 仅从 Room 流读取状态，不从 Compose 内存恢复任务。
- 多会话同时运行触发服务商限流时，仅对应任务进入失败/重试状态。

## 测试与验收

### 单元测试

- 入队后草稿立即清空，入队失败时保留草稿。
- 每会话 FIFO、跨会话并行和单会话至多一个运行任务。
- 队列项编辑、删除、引导提升及其顺序保持。
- 引导当前时旧助手消息保留且标记正确，新任务优先执行。
- 进程恢复时 `RUNNING` 转为 `INTERRUPTED`，`QUEUED` 保留。
- 中文偏好注入不覆盖明确的用户语言要求。
- 思考摘要选取最新非空行、单行省略状态和展开状态。

### Android 测试

- 发送后立即清空文字和单图预览。
- 生成中继续输入并排队，当前完成后自动执行下一项。
- 切换会话后两个会话可同时流式更新。
- 切到后台、锁屏再返回后流式内容持续落库并可见。
- 队列项引导当前、编辑、删除的交互和无障碍内容描述。
- 前台通知在有活动流时存在、所有任务空闲后消失。

### 手工验收

1. 在同一会话连续发送三条消息，确认后两条等待并按顺序执行。
2. 在流式中选中第二条等待消息执行“引导当前”，确认旧回复保留且新回复优先开始。
3. 同时在两个会话发起流式请求，确认两者并行更新。
4. 发送后切到其他应用、锁屏再返回，确认回复不中断。
5. 检查思考过程默认显示最新一行中文摘要，点击后可展开完整内容。
