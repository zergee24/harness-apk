# Harness APK 温暖无障碍核心界面设计

## 结论

本轮采用“温暖实用型 Material 3”优化 Harness APK 的共享主题、首页、聊天页和项目工作台。目标用户包含老人和女性用户，界面需要同时具备亲和力、可读性、明确反馈和开发工具应有的专业感。

设计不使用粉色化或装饰化表达来定义女性用户，也不通过低对比度、过度留白或小字号营造所谓精致感。温度来自暖色主调、自然文案、舒适间距、清晰层级和可预期交互。

本轮只调整 Compose 视觉系统、布局和交互呈现，不改变现有导航结构、业务行为、数据模型、状态管理和功能入口含义。

## 设计判断

页面类型：原生 Android 个人 AI 助手和移动项目工作台。

目标受众：需要低学习成本的老人用户、重视舒适度与亲和力的女性用户，以及现有开发工具用户。

视觉语言：温暖、清晰、可信赖的 Material 3，采用更鲜明但不过度饱和的主色，以及稳定、低噪声的生产力工具布局。

设计参数：

- `DESIGN_VARIANCE: 4`：布局稳定，允许轻微层次变化，不做实验性排版。
- `MOTION_INTENSITY: 3`：仅保留点击反馈、状态切换和必要过渡。
- `VISUAL_DENSITY: 4`：信息完整，但通过分组和间距避免拥挤。

## 当前问题

当前应用已经使用 Jetpack Compose 和 Material 3，但视觉层仍偏向框架默认值：

- 主题只定义基础颜色和圆角，未建立明确的字体、间距、容器和状态规范。
- 首页会话列表大量依赖悬浮卡片，扫描效率和信息层级不足。
- 聊天页功能较多，消息、模型配置、上下文状态和输入操作容易争夺注意力。
- 项目工作台同时承载项目选择、会话、文件和 Git，卡片嵌套和动作密度容易造成压迫感。
- 不同页面重复使用零散的 `dp`、圆角和透明度，导致视觉节奏不一致。
- 小字号、次级文字对比度和紧凑触控区域对老人用户不够友好。

## 目标

1. 建立统一的暖色 Material 3 主题和共享视觉规范。
2. 让正文、说明、状态和主要操作形成清晰层级。
3. 保证主要触控区域适合手部精细操作能力较弱的用户。
4. 降低首页、聊天页和项目工作台的卡片感与视觉噪声。
5. 保持浅色和深色模式中的品牌识别、对比度和语义一致。
6. 保留现有功能、导航位置、数据流和用户操作习惯。

## 非目标

- 不改变路由、一级模式或页面信息架构。
- 不增删聊天、项目、文件、Git 或模型配置功能。
- 不修改数据库、Repository、网络请求和状态模型。
- 不重写现有业务文案，只允许为清晰度调整局部提示。
- 不引入新的第三方 UI 框架。
- 不加入持续动画、玻璃拟态、渐变发光和装饰性插画。
- 不在本轮全面改造设置及其所有二级页面。

## 全局视觉系统

### 色彩

主色统一采用低饱和珊瑚色相，用于主要按钮、选中状态、焦点和关键进度。浅色模式使用略深的浅珊瑚色 `#D98278`，主色文字使用 `#5F1410`，主色容器使用 `#FFDAD5`；深色模式使用更亮的浅珊瑚色 `#FFB3AA`，主色文字同样使用 `#5F1410`。两种模式保持同一色相，通过明度差保证层级和对比度。

浅色模式背景使用 `#FAF7F6`，内容表面使用 `#FFFDFC`，正文使用 `#211A19`，次级容器使用 `#F1E5E2`。深色模式背景使用 `#17181A`，内容表面使用 `#202124`，正文使用 `#F3EDEC`，次级容器使用 `#343437`，避免纯黑背景造成强烈反差。

成功、警告和错误使用独立语义色，只在真实状态中出现。次级信息使用中性灰蓝，不使用彩色圆点或多种强调色进行装饰。

上述色值通过 Material 3 `ColorScheme` 集中定义。实施时只允许在对比度检查不达标的情况下微调明度，并需要保持珊瑚色系的品牌识别。全应用只保留珊瑚色及其明度变体作为品牌强调色。

### 字体

继续使用 Android 系统字体和中文系统回退，避免新增字体包带来的体积、渲染和中文覆盖问题。

建议字体层级：

- 页面标题：`22sp` 左右，中等或半粗字重。
- 区块标题：`18sp` 左右，半粗字重。
- 正文：`17sp` 左右，常规字重，行高约 `25-27sp`。
- 说明文字：不低于 `14sp`，行高约 `20sp`。
- 操作标签：`15-16sp`，中等字重。
- 时间和非关键元数据：允许 `13-14sp`，不得承载必要操作信息。

文字必须支持系统字体缩放。重要按钮、标题和状态不得依赖固定高度，以免放大字体后截断。

### 形状与层级

采用一致的中等圆角体系：

- 小型控件和标签：`10dp`。
- 输入框、按钮和列表容器：`14dp`。
- 对话框、空状态和重要面板：`18dp`。

页面不使用卡片套卡片。优先通过背景层级、分组间距和单条分隔线组织内容。阴影只用于顶部栏、底部输入面板、弹层等确实需要浮层关系的位置。

### 间距与触控

共享间距以 `4dp` 为基础，主要使用 `8dp`、`12dp`、`16dp`、`20dp` 和 `24dp`。

- 页面水平边距：手机默认 `16dp`，宽屏可增加到 `24dp`。
- 区块间距：`20-24dp`。
- 列表行内部垂直间距：`12-16dp`。
- 主要触控区域：最小 `48dp`。
- 主要按钮和输入区域：优先 `52-56dp`。
- 图标按钮图标：常规 `22-24dp`，危险操作必须带文本确认。

### 动效与反馈

动效只服务于反馈和状态变化：

- 按钮按下使用轻微缩放或颜色变化。
- 分段控件切换使用短距离颜色和位置过渡。
- 空状态、加载状态和错误状态采用淡入，不使用循环动画。
- 页面滚动、消息流和输入框不加入装饰性运动。
- 尊重系统减少动态效果设置，必要时直接切换状态。

## 首页设计

### 顶部区域

“会话 / 项目”继续位于顶部栏，但改为边界清晰、文字更大的分段控件。当前模式使用暖莓红容器和高对比文字，未选模式保持中性背景。

新建和设置保留图标按钮。更新提示继续使用真实语义标记，但扩大可点击区域，避免只依赖一个小圆点表达状态。

### 会话列表

会话列表从每项独立悬浮卡片调整为平面列表：

- 标题是第一视觉层级，最多两行。
- 更新时间和所属项目位于标题下方，保持较低但合格的对比度。
- 项目归属使用文字标签或轻量背景，不使用强色实心胶囊。
- 更多操作保持在行尾，点击区域不少于 `48dp`。
- 行与行之间使用间距和单条浅分隔线，不增加重复阴影。

项目分组模式保留现有折叠逻辑，分组标题需要显示项目名称、会话数量和明确的展开状态。

### 空状态

空状态使用暖色浅容器、清晰标题、短说明和“新建会话”主要按钮。说明只保留本地保存与记忆摘要的关键信息，不把操作说明写成大段帮助文字。

## 聊天页设计

### 消息区

助手消息默认无气泡或使用透明容器，使 Markdown 内容成为主体。用户消息使用淡暖莓红背景和深色文字，保证角色区分但不过度占据视觉空间。

系统事件、错误、文件变更和工具结果继续使用专用组件，不强行套入普通消息气泡。代码、表格和 diff 保留自身可滚动和等宽字体规则。

消息时间、模型和状态等元数据放在消息底部的次级行，默认弱化。失败和停止状态使用文字加图标，不只依赖颜色。

### 输入区

输入区改为稳定的底部操作面板：

- 文本输入是主区域，默认高度舒适，支持多行增长。
- 发送按钮使用高对比暖莓红主色。
- 语音、附件和停止使用标准图标按钮，并提供内容描述。
- 文件变更模式等重要模式必须有清晰文字状态，不能只换图标颜色。
- 键盘弹出后输入区保持完整可见，按钮不遮挡文本。

### 会话状态

模型、推理强度、联网和上下文状态集中在输入区上方或内部的一行次级信息。只有当前可操作或需要注意的状态常驻，完整配置继续通过会话设置打开。

上下文压力、生成中、停止中和错误状态使用明确短句。进度不得只通过细小轨道或难以分辨的颜色表达。

## 项目工作台设计

### 顶部项目区

当前项目名称是第一层级。项目切换和新建会话作为直接动作，克隆、导入、导出、分享、删除等低频动作收进清晰菜单。

危险操作继续通过确认对话框完成。删除入口不得与新建、保存等常用动作并列为同等强调。

### 标签页

“会话 / 文件夹 / Git”保留现有结构，使用固定高度、可滚动或等宽分配的 Material 3 标签页。当前标签使用主色文字和明确指示器，不能只依赖细小颜色变化。

### 会话标签页

项目会话列表复用首页列表语言。新建项目会话是主要按钮，已有会话按更新时间组织。空状态直接提供新建入口。

### 文件夹标签页

搜索、筛选、目录树和内容预览建立明确顺序：

1. 搜索与筛选。
2. 文件目录和当前选中项。
3. 预览或编辑内容。
4. 保存、打开、分享和导出动作。

手机窄屏保持单列，目录与内容通过选择状态切换，不在同一屏硬塞双栏。大屏可以使用主从双栏，但必须保持触控尺寸和文本缩放能力。

### Git 标签页

首先展示当前分支、工作区状态和领先/落后信息。刷新等常用安全动作直接显示，提交、拉取、切换分支和推送按风险分组。

冲突、脏工作区阻断和远端缺失等错误必须在相关操作附近显示，并说明下一步。危险或不可恢复动作不使用主色强调。

## 共享组件

本轮只抽取能明显减少视觉漂移的组件：

- `WarmSegmentedControl`：首页模式切换。
- `ComfortListRow`：会话、项目会话和轻量设置行。
- `SectionHeader`：统一区块标题与可选动作。
- `ActionableEmptyState`：带主操作的空状态。
- `InlineStatusMessage`：加载、成功、警告和错误状态。
- `ChatComposerSurface`：聊天底部输入面板。

共享组件只接收展示状态和回调，不持有 Repository、导航或业务协程。页面继续负责数据订阅和事件处理。

## 状态和数据流

现有数据流保持不变：

1. 页面从 Repository 或现有 UI 状态收集数据。
2. 页面将纯展示数据传给共享组件。
3. 共享组件通过回调上报点击、输入和切换事件。
4. 页面调用现有业务方法并更新状态。
5. 加载、空、错误和成功状态映射为统一视觉组件。

视觉重构不得把业务逻辑移动到主题或共享组件中，也不得为了动画引入额外全局状态。

## 错误、加载和空状态

- 加载：优先使用与最终布局一致的占位结构；短操作可以使用按钮内进度和禁用状态。
- 空状态：必须说明当前为空的对象，并提供一个最合理的下一步操作。
- 错误：在发生错误的区域附近显示，包含简短原因和可执行的重试或返回动作。
- 成功：对保存、导入、导出等瞬时结果使用短状态提示，不长期占据页面。
- 无障碍：所有状态同时使用文字、图标或结构表达，不只依赖颜色。

## 测试与验收

### 自动化验证

- 原有 JVM 单元测试继续通过。
- 为主题令牌和关键展示状态补充可测试的纯数据映射或 Compose 测试。
- 对首页分段控件、空状态操作、聊天输入状态和项目标签页进行关键交互测试。
- 执行 `git diff --check`，避免格式问题。

### 模拟器验证

至少验证以下组合：

- 浅色模式与深色模式。
- 默认字体与系统放大字体。
- 常规手机宽度与窄屏手机宽度。
- 空会话、长标题、多项目分组和长会话列表。
- 键盘弹出、多行输入、发送中、停止中和错误状态。
- 项目为空、文件较多、长文件名、Git 干净和 Git 有变更状态。

### 验收标准

- 主要触控目标不小于 `48dp`。
- 正文默认约 `17sp`，必要说明不小于 `14sp`。
- 字体放大后主要标题、按钮和输入区不被裁切。
- 浅色与深色模式中，主要文字和按钮达到 WCAG AA 对比度目标。
- 首页、聊天页和项目工作台使用同一套颜色、圆角、间距和状态表达。
- 不出现卡片套卡片、无意义阴影和多种装饰性强调色。
- 键盘不遮挡输入内容和发送操作。
- 长会话和项目文件列表滚动稳定。
- 原有功能入口、导航和业务行为保持可用。

## 实施顺序

1. 定义主题色、Typography、Shapes 和共享尺寸令牌。
2. 实现共享分段控件、列表行、空状态和状态提示。
3. 优化首页和会话列表。
4. 优化聊天消息区、状态区和底部输入面板。
5. 优化项目工作台顶部区、标签页和三个内容状态。
6. 补充自动化测试。
7. 在模拟器完成浅色、深色、默认字体和放大字体视觉验收。

## 风险与控制

- 字体放大导致布局溢出：避免固定文本高度，关键按钮允许宽度扩展，窄屏操作改为菜单。
- 主题变化影响 Markdown 和代码对比度：单独检查代码块、表格、diff 和链接颜色。
- 共享组件抽取过度：只抽取跨两个以上页面且规则稳定的视觉组件。
- 项目工作台改动范围过大：保留标签页和业务结构，只逐层替换视觉容器。
- 动效影响低端设备：使用 Compose 原生短过渡，并支持减少动态效果。

## 设计自检结论

- 文档没有未决占位符或开放项。
- 视觉系统、页面设计、实现边界和验收标准一致。
- 范围限定在共享主题、首页、聊天页和项目工作台，可由单个实施计划完成。
- “有温度”已明确为色彩、排版、间距、反馈和可访问性，不依赖性别刻板印象。
