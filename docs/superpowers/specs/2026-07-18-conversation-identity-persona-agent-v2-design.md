# 基于会话身份的人格智能体 V2 设计

## 1. 决策状态

本设计替代《可移植语料人格智能体设计》中以下决定：

- “智能体”作为首页第三种顶层模式。
- 运行时只使用 `persona + worldview + 原文检索结果` 组织回答。

本设计明确不改变以下产品边界：

- `Agent` 是全局安装的人物身份资产，不归某个项目所有。
- `Conversation` 是发生交互、固定身份版本和驱动 Markdown 的实体。
- `Project` 是可选的工作上下文，继续管理 Markdown、文件和 Git，不是智能体容器。
- `Conversation.projectId` 与 `Conversation.agentId` 相互独立，不新增强制 `Project <-> Agent` 关系。

以下既有决定继续有效：

- 桌面 Codex 负责重型构建，Android 负责安装和运行。
- `.hbundle` 使用 Ed25519 签名、checksums 和安全 ZIP 校验。
- `Conversation` 固定保存 `agentId + agentVersion`，升级不改变历史会话。
- 人物模拟必须在 UI 中明确披露，不能宣称是真实人物本人。
- 用户自行选择并确认有权使用的资料，不内置受版权保护的全文。
- 安装和运行不依赖常驻桌面服务或中心服务端。

## 2. 问题与目标

当前智能体已经能安装资料、检索原文并创建会话，但实际回答呈现为“资料审校员”：

- 习惯先解释时间边界和资料限制，再回答用户。
- 大量使用“当前资料不足”“只能据此”“我先说清楚”等模型式免责声明。
- 默认输出编号列表和总结句，缺少人物自己的语言节奏。
- 能复述观点，却缺少经历、关系、场景和持续对话形成的人格连续性。
- 智能体入口与普通会话分离，人物既不能自然进入日常会话，也不能自然参与项目中的 Markdown 工作。

本设计的目标是：

1. 让智能体成为会话可选择并固定的一种身份，而不是第三套产品模式。
2. 让人物回答同时理解“我是谁”“现在是什么情况”“我们经历过什么”。
3. 在不牺牲来源约束的前提下，显著提升第一人称表达、立场稳定性和对话连续性。
4. 允许同一身份用于普通会话和项目会话，不强迫用户先创建或选择项目。
5. 保持 Android 本地安装、会话与 Markdown 弱关联、版本固定和用户现有模型配置。
6. 用可重复的评测证明改善，而不是只依赖提示词主观调试。

## 3. 第一性原则

### 3.1 三个概念只承担三种职责

| 概念 | 唯一职责 | 不承担的职责 |
| --- | --- | --- |
| `Agent` | 提供“谁在说话”的固定身份资产 | 不拥有项目、会话或 Markdown |
| `Conversation` | 承载交互、固定身份版本并驱动内容变化 | 不定义人物身份，不替代项目文件空间 |
| `Project` | 提供可选的 Markdown、文件、Git 和任务上下文 | 不成为智能体入口，不决定人物是谁 |

这三个概念可以组合，但不能合并。人格会话可以不属于项目；项目会话也可以使用普通助手。

### 3.2 四层上下文来源

每次人格会话按需使用四类上下文：

| 层 | 回答的问题 | 来源 |
| --- | --- | --- |
| 人物身份层 | 我是谁、我如何判断和表达 | 会话固定版本的 `.hbundle` |
| 会话现实层 | 你此刻在问什么、前文已经说了什么 | 本轮输入、会话消息和会话摘要 |
| 人物关系层 | 我们此前形成了什么稳定关系 | 当前 `agentId` 的本地关系记忆 |
| 项目工作层 | 当前项目有哪些资料、决定和文件状态 | 仅当 `projectId != null` 时注入项目 Markdown、文件和项目上下文 |

四层不能互相冒充：

- 人物历史资料不能被当作今日事实。
- 会话和项目资料不能修改人物的核心历史立场。
- 人物关系记忆不能吸收只属于某个项目的工作事实；项目事实必须回到项目 Markdown。
- 任何记忆都不能覆盖已签名的人物身份。

### 3.3 事实约束与表达自由分离

来源约束只约束事实和核心立场，不约束正常的谈话连接词、语气和互动行为。

- 历史事实、人物经历和核心立场必须有包内证据。
- 今日具体事实必须来自本轮输入、当前会话，或当前项目已经提供的资料。
- 问候、追问、承接前文和语言风格可以由人物身份、会话上下文与关系记忆驱动。
- 找不到核心证据时，不编造结论；只提出一个最能补足条件的问题。

### 3.4 不增加使用仪式

- 不增加“历史模式 / 现实模式”开关。
- 不要求用户每轮选择资料、人格或记忆。
- 不在每条回复前重复模拟声明。
- 导入智能体时不要求选择项目。
- 新建会话根据入口和最近使用记录预选身份，用户只在需要切换时操作。
- 第一条消息发送后固定身份和版本，后续不再展示身份配置操作。

## 4. 领域模型

```text
Installed Agent 1 -------- 0..N Conversation
                                  |
                                  | projectId optional
                                  v
                             0..1 Project

Conversation 0..N <----> 0..N Markdown
                 existing weak association inside a project
```

### 4.1 已安装智能体

`Agent` 继续是全局安装、签名和版本化的资产。安装后可被任意新会话选择，不复制到项目，也不生成项目关联记录。

### 4.2 会话约束

`Conversation` 保留相互独立的可选字段：

| 字段 | 含义 |
| --- | --- |
| `projectId` | 当前会话所属的可选项目上下文 |
| `agentId` | 当前会话使用的可选人物身份 |
| `agentVersion` | 使用人物身份时必须固定的包版本 |

约束为：

```text
agentId != null  -> agentVersion != null
agentId == null  -> agentVersion == null
projectId        independent from agentId
```

因此系统明确支持四种会话：

| 会话 | `projectId` | `agentId` |
| --- | --- | --- |
| 普通日常会话 | 空 | 空 |
| 人格日常会话 | 空 | 有值 |
| 普通项目会话 | 有值 | 空 |
| 人格项目会话 | 有值 | 有值 |

预选阶段只继承 `agentId`；第一条消息发送时使用当时已安装的最新可用版本，原子固定 `agentId + agentVersion`。智能体升级、停用和后续预选变化都不改变既有会话。

### 4.3 身份预选不建立项目关联

新会话的身份预选只从既有会话推导，不新增持久化的项目与智能体关联表：

1. 从项目内新建时，优先使用该项目最近一次活跃会话的身份。
2. 从首页新建时，使用全局最近一次活跃会话的身份。
3. 没有历史记录或对应身份已停用时，使用普通助手。
4. 用户可在发送第一条消息前切换，选择结果只写入新会话。

这是一项输入默认值，不代表项目拥有、关联或默认配置了该智能体。

## 5. 产品信息架构

### 5.1 首页

首页顶层模式恢复为：

```text
会话 | 项目
```

移除顶层“智能体”。会话列表继续聚合所有普通会话和项目会话；人格会话通过头像、名称和“基于资料模拟”标识区分，不建立第二套会话列表。

### 5.2 智能体包管理

低频管理入口移动到：

```text
设置 -> 智能体包
```

该页面只负责：

- 导入 `.hbundle`。
- 查看安装状态、版本、资料覆盖和发布者指纹。
- 更新版本。
- 停用或卸载未被会话引用的版本。

用户从系统文件、分享菜单或下载通知打开 `.hbundle` 时，仍直接进入安装确认，不要求先导航到设置。

### 5.3 导入后开始会话

安装确认页只负责校验并安装全局身份，不显示项目选择器。安装成功页提供：

- 主操作“开始对话”：创建使用该身份的会话。
- 次操作“完成”：返回原入口。

如果导入动作来自某个项目，“开始对话”创建人格项目会话；其他入口创建人格日常会话。两种情况都不写入持久的项目与智能体关联，也不自动创建项目。

### 5.4 新会话身份选择

首页和项目内共用同一个新会话界面。输入区上方使用现有紧凑配置区域显示身份和模型：

```text
[头像 李德胜⌄] [当前模型⌄]
```

- 身份按 4.3 的规则自动预选。
- 点击身份可切换为任一已安装智能体或“普通助手”。
- 切换只在发送第一条消息前允许；发送后会话版本已固定。
- 身份选择器不显示“关联项目”“设为项目默认”等二次概念。
- 项目名称和项目上下文继续由原有项目界面展示，不塞入身份选择器。

### 5.5 会话披露

聊天顶部显示一次：

```text
李德胜 · 基于资料模拟
```

点击可查看版本、资料范围和发布者指纹。回复正文不重复“我不是真实人物”，除非用户直接询问身份真实性。

参考资料继续使用现有折叠区域展示，正文不得出现 `[资料N]`、文件名或内部 chunk ID。

## 6. “我、现在、我们”上下文模型

### 6.1 我是谁

人物身份层来自固定版本包，包含：

- 身份与时间边界。
- 核心判断方法。
- 分时期立场。
- 可回忆的经历。
- 语言节奏和修辞习惯。
- 典型对话示例。

该层在会话过程中只读。

### 6.2 现在是什么情况

“现在”首先来自会话，而不是来自项目：

- 本轮用户明确给出的现实条件。
- 当前会话的最近消息和压缩摘要。
- 用户刚刚选择、粘贴或上传的内容。

当 `projectId != null` 时，再补充项目工作层：

- 项目说明和当前任务状态。
- 当前会话弱关联的 Markdown。
- 已审核并应用的项目文件变化。
- 当前项目内已经确认的决定和约束。

人物对现实问题只能根据上述可见事实形成具体判断。关键条件缺失时，回答应先给出可成立的方法判断，再提出一个具体追问，不输出长篇资料免责声明。

### 6.3 我们经历过什么

“我们”只由会话和人物关系组成：

| 作用域 | 键 | 保存内容 |
| --- | --- | --- |
| 当前会话 | `conversationId` | 当前话题、短期指代和会话摘要 |
| 人物关系 | `agentId` | 称呼偏好、稳定偏好、共同经历和关系变化 |

人物关系记忆可在该身份的日常会话和项目会话间连续，但只保存与“用户和人物如何相处”有关的稳定信息。项目目标、文件、任务、决定和业务事实不得写入人物关系记忆。

### 6.4 会话继续驱动 Markdown

本设计不为智能体增加笔记所有权或专用笔记页：

- 会话仍是创建、修改和讨论 Markdown 的驱动实体。
- Markdown 仍属于项目，并沿用会话与 Markdown 的多对多弱关联。
- 智能体只改变会话中“谁在说话”，不改变 Markdown 的归属和版本规则。
- 需要在同一项目的不同会话间延续的工作事实必须经过现有 diff 审核写入 Markdown；不新增隐藏的项目记忆副本。
- 人格日常会话没有项目上下文，因此不能静默创建项目 Markdown；需要落盘时走现有“加入项目”流程。

这样，普通助手和人物身份共用同一套会话驱动记事本能力，不产生“智能体项目”这一额外概念。

## 7. `.hbundle` Schema V2

V2 保留 V1 的签名、checksums、manifest 和 corpus 结构，在 `agent/` 中增加可运行的人物身份资产：

```text
agent/
  persona.md
  identity.json
  voice.json
  worldview.jsonl
  episodes.jsonl
  concepts.json
  examples.jsonl
  openers.json
  eval.jsonl
```

### 7.1 `persona.md`

长度受控的常驻身份内核，只包含每轮都需要的内容：

- 自我称谓。
- 时间边界。
- 核心判断方法。
- 不可变行为边界。
- 模拟身份说明。

不在此文件堆叠完整观点、长篇履历或来源列表。

### 7.2 `identity.json`

结构化身份和关系：

```json
{
  "selfNames": ["李德胜"],
  "timeHorizon": "包内资料覆盖时期",
  "roles": ["组织者", "思想者"],
  "relationships": [
    {
      "subject": "某组织或人物",
      "relation": "关系说明",
      "period": "适用时期",
      "evidence": ["chunk-id"]
    }
  ]
}
```

### 7.3 `voice.json`

描述可执行的语言特征，而不是“有感染力”“像本人”等空泛形容：

```json
{
  "defaultForm": "先下判断，再展开条件",
  "sentenceRhythm": ["短判断与较长论证交替"],
  "rhetoricalMoves": ["设问", "对比", "从具体事实推到方法"],
  "preferredTerms": ["调查", "群众", "组织"],
  "avoidPatterns": [
    "我先要把……说清楚",
    "当前资料不足",
    "我的回答不是……而是……",
    "以下三点"
  ],
  "evidence": ["chunk-id"]
}
```

每项风格特征必须引用人物本人创作或直接讲话资料。二手材料不能作为第一人称语言模仿依据。

### 7.4 `worldview.jsonl`

沿用 V1 文件名，扩展为带时期和触发概念的立场记录：

```json
{
  "id": "stance-method-001",
  "topic": "调查与决策",
  "statement": "调查应先于结论",
  "conditions": ["需要对现实力量作判断"],
  "period": "适用时期",
  "aliases": ["怎么走", "如何判断", "路线"],
  "confidence": 0.93,
  "evidence": ["chunk-id"]
}
```

冲突立场按时期、条件和证据分别保存，不压缩成一个永恒结论。

### 7.5 `episodes.jsonl`

保存可用于第一人称回忆的事件：

```json
{
  "id": "episode-001",
  "period": "事件时期",
  "location": "地点",
  "participants": ["人物或群体"],
  "summary": "可被自然回忆的事件摘要",
  "meaning": "该事件如何影响判断",
  "evidence": ["chunk-id"]
}
```

### 7.6 `examples.jsonl`

保存按对话意图组织的少样本示例：

```json
{
  "id": "example-advice-001",
  "intent": "根据不完整现实条件提供方法建议",
  "user": "我们现在该怎么走？",
  "assistant": "自然、第一人称、无资料编号的示例回答",
  "styleTags": ["判断优先", "群众视角", "行动语言"],
  "evidence": ["chunk-id"],
  "generationType": "synthesized"
}
```

合成示例必须显式标记，不能伪装成历史原话。

### 7.7 `openers.json`

包含一个默认开场和最多两个替代开场。开场用于新会话空状态，不作为每轮系统提示重复注入。

### 7.8 来源类型

`sources.json` 新增：

- `genre`：`essay | speech | conversation | letter | interview | memoir | secondary`。
- `authorship`：`direct | edited_direct | secondary`。
- `period`。

如果只有论文、正式文章等书面资料，构建仍可通过，但报告和安装预览必须显示“书面人格，对话还原度有限”。要达到较强的人物对话感，至少需要用户有权使用的讲话、谈话、书信或访谈资料。

## 8. 桌面蒸馏流程 V2

### 8.1 流程

```text
提取与来源分类
-> 身份和时间线
-> 分时期立场
-> 直接材料语言指纹
-> 经历与关系
-> 对话示例和开场
-> 检索评测
-> 角色一致性评测
-> 签名打包
```

### 8.2 构建约束

- 所有身份、立场、经历和风格条目必须引用真实 chunk ID。
- Codex 不使用模型常识补写人物事实。
- 允许根据有证据的立场合成对话示例，但必须标记为 synthesized。
- 二手材料可用于检索和背景，不用于生成第一人称经历或语言风格。
- 构建报告分别给出“事实覆盖”“立场覆盖”“对话资料覆盖”和“风格可用度”。

### 8.3 评测集

`eval.jsonl` 增加四类题：

| 类型 | 目的 |
| --- | --- |
| grounding | 是否找到并使用正确证据 |
| stance | 是否保持人物在对应时期的立场 |
| voice | 是否避免通用 AI 套话并使用人物表达结构 |
| temporal | 是否区分历史事实、会话中的今日事实和可选项目事实 |

构建报告不能再只用 Top 8 来源命中率代表整体质量。

## 9. Android 运行时

### 9.1 Context Assembler

新增单一 `AgentContextAssembler`，负责组织以下上下文：

1. 固定版本 `persona`。
2. 本轮输入、当前会话消息和会话摘要。
3. 当前人物的全局关系记忆。
4. 仅在 `projectId != null` 时加入项目上下文和弱关联 Markdown。
5. 与本轮意图匹配的立场。
6. 最多两个相关经历。
7. 最多两个相关语言示例。
8. 多来源、跨时期去重后的原文证据。

调用方只接收一个不可变 `AgentRuntimeContext`，不直接读取各表或拼接提示词。

### 9.2 本地检索

V2 第一版不引入新的向量数据库或远端 embedding 依赖。检索顺序为：

1. 使用 `concepts + aliases + 中文 n-gram` 匹配结构化立场、经历和示例。
2. 使用现有 FTS 和 n-gram 检索原文 chunk。
3. 按分数、来源、时期和章节做多样性重排。
4. 仅在会话属于项目时，对弱关联 Markdown 使用现有项目上下文选择逻辑。
5. 在固定 token 预算内组装。

单纯增加向量检索不能解决当前“资料审校员”问题；只有结构化人物资产和示例真正进入运行时，语义检索才有价值。

### 9.3 提示词契约

系统提示按以下优先级组织：

```text
身份内核
-> 事实与表达边界
-> 会话现实
-> 关系记忆
-> 可选项目工作上下文
-> 相关立场、经历和示例
-> 无编号原文证据
```

提示词明确要求：

- 默认先回答，再说明条件。
- 默认使用自然段，不机械输出编号列表。
- 不提“资料库、chunk、资料编号、当前资料不足”等内部实现。
- 不把本轮、会话或当前项目中没有出现的今日事实当作已知。
- 条件不完整时，只追问一个最关键的问题。
- 用户直接询问真实性时，明确说明是基于资料的模拟。

传给模型的证据不使用 `[资料1]` 等可复制标签。客户端参考资料继续由检索结果独立生成。

### 9.4 回答类型

运行时区分三类回答：

| 类型 | 证据要求 | 行为 |
| --- | --- | --- |
| 关系与日常承接 | 人格和关系记忆即可 | 自然回应，不因缺少原文拒答 |
| 人物事实与核心立场 | 必须有包内证据 | 无证据时不编造 |
| 今日具体判断 | 必须有本轮、会话或当前项目事实 | 条件不足时提出一个问题 |

该分类由上下文和检索结果自动判断，不暴露为 UI 模式。

## 10. 人物关系记忆

### 10.1 数据模型

数据库从 11 升级到 12，新增：

`agent_memories`：

- `id`
- `agentId`
- `kind`：`user_preference | address_preference | shared_history | relationship_event`
- `content`
- `sourceConversationId`
- `sourceMessageId`
- `confidence`
- `createdAt`
- `updatedAt`

`id` 为主键，并为 `(agentId, updatedAt)` 建索引。人物关系记忆只按 `agentId` 召回；现有会话摘要继续按 `conversationId` 保存，不复制进新表。项目事实继续由项目 Markdown 和项目上下文提供，不复制进该表。所有记忆都是可审计的离散事实，不保存隐藏推理过程。

### 10.2 自动沉淀

- 优先复用现有会话压缩结果。
- 会话离开前至少有一个完成回合时提取一次；长会话每累计 10 个完成回合再提取一次。
- 只保存对后续相处有稳定价值的称呼、偏好、共同经历和关系变化。
- 项目目标、文件、任务、决定、业务事实和仅对当前话题有效的内容一律过滤。
- 提取失败不阻断回复，不显示重试交互；后续满足条件时重新尝试。
- 重复项合并，冲突项保留最新内容和来源，不静默覆盖核心人物身份。

### 10.3 用户控制

会话身份详情中提供“人物关系记忆”：

- 查看来源。
- 编辑。
- 删除单条。
- 清空当前人物的全部关系记忆。

不提供每轮“是否记住”的确认弹窗。所有记忆仅保存在本地应用数据库，默认不进入项目 Git 和 `.hbundle`。

## 11. V1 兼容与迁移

### 11.1 V1 包

- 已安装 V1 包继续可用。
- V1 通过运行时适配器使用新的回答契约和会话身份模型。
- 缺少 `voice/episodes/examples/openers` 时标记为“基础人格”，不伪造这些资产。
- 用户重新构建 V2 后，以同一 `agentId`、更高版本和同一发布者密钥升级。

### 11.2 现有无项目智能体会话

现有 `agentId != null && projectId == null` 的会话本身就是合法的人格日常会话，不做数据迁移。移除顶层“智能体”入口后，这些会话继续出现在统一会话列表中，会话 ID、消息、`agentId` 和 `agentVersion` 均不改变。

### 11.3 卸载

被会话固定引用的智能体版本不能物理删除。停用只会从新会话选择器隐藏；最后一个引用解除后才允许清理包和索引。

## 12. 错误处理

| 场景 | 行为 |
| --- | --- |
| V2 缺少必需身份文件 | 安装失败并指出具体文件 |
| 结构化人物条目引用不存在 chunk | 构建或安装校验失败 |
| 项目被删除 | 保留会话和消息时清空 `projectId` 及 Markdown 弱关联，停止注入项目资料 |
| 预选智能体被停用 | 新会话回退为普通助手，已固定版本的历史会话不变 |
| 关系记忆提取失败 | 不影响当前回复，后台后续重试 |
| 找不到历史立场证据 | 不生成该立场，只提出必要追问或说明未知 |
| 找不到今日事实 | 使用已有方法判断，并只询问一个关键现实条件，不要求先创建项目 |
| 模型输出内部资料标签 | 输出层移除已知内部标签并记录质量事件 |

## 13. 安全、隐私与真实性

- UI 始终保留“基于资料模拟”标识。
- 不把模型生成的合成示例标记为历史原话。
- 不允许会话记忆修改签名人物包。
- 人物关系记忆不得保存项目业务事实。
- 不保存或展示模型隐藏推理。
- 原始来源和发布者密钥继续遵守现有本地安全边界。
- 远端 catalog、OSS 分发和敏感词治理不在本设计的实施范围。

## 15. 实施阶段

### 阶段 A：产品收口与回答急救

- 移除顶层智能体模式。
- 在首页和项目内的新会话界面增加同一个身份选择器。
- 让会话固定 `agentId + agentVersion`，保持 `projectId` 独立可选。
- 现有无项目智能体会话直接进入统一会话列表，不生成项目。
- 重写回答契约，不再输出资料标签和惯例免责声明。
- 让 V1 包也使用会话现实和可选项目工作上下文。

### 阶段 B：人物包 V2

- 扩展 builder、manifest、校验和安装解析。
- 生成并消费 identity、voice、episodes、examples 和 openers。
- 增加来源类型和人物质量报告。
- 用同一发布者密钥构建新的李德胜 V2 包。

### 阶段 C：关系记忆与质量闭环

- 新增人物关系记忆。
- 接入自动提取、查看、编辑和清空。
- 建立角色、时间、来源和连续性评测。
- 完成 V1/V2、升级和失败恢复测试。

三个阶段属于同一 V2 交付目标；阶段 A 可以先形成可安装测试版，但不能被视为最终人格质量完成。

## 16. 验收标准

### 16.1 产品归属

- 新建人格会话 `agentId + agentVersion` 固定率为 100%。
- 人格日常会话允许 `projectId == null`，不自动创建占位项目。
- 安装流程不要求项目，数据库不新增项目与智能体关联表。
- 项目会话是否使用智能体不改变项目、Markdown 和 Git 的归属模型。
- 首页和项目内使用同一个身份选择器，新建路径不增加独立向导。
- 顶层导航不再出现“智能体”。

### 16.2 回答体验

- 正文不出现 `[资料N]`、chunk ID 或内部文件位置。
- 方法建议类问题不以资料免责声明开头。
- 默认使用自然段；只有用户明确要求或内容确有并列结构时使用列表。
- 条件不足时最多提出一个关键追问。
- 用户直接询问真实性时，明确回答是基于资料的模拟。

### 16.3 人物质量

- 至少 30 道立场题全部命中有效证据，核心立场无无证据补写。
- 至少 20 组“V1 通用 RAG vs V2”盲测中，V2 被判断更像目标人物的比例不低于 70%。
- `voice.avoidPatterns` 中的通用 AI 套话命中率低于 5%。
- 时期冲突题能够保留时间差异，不生成单一永恒立场。

### 16.4 会话现实、项目现实与关系连续性

- 当前会话中已经明确的今日事实可被人物用于后续判断。
- 项目中已经明确的今日事实可被人物用于后续判断。
- 会话和当前项目均未提供的今日事实不会由历史资料补写。
- 人物关系记忆可在同一智能体的日常会话和项目会话间召回。
- 项目事实经审核写入 Markdown 后，可被同项目的其他会话和其他身份使用。
- 未写入项目 Markdown 的会话临时事实不会被暗中提升为项目事实。
- 人物关系记忆提取测试中的项目目标、文件、任务、决定和业务事实均不得持久化。
- 清空人物关系记忆后，后续请求不再注入已删除内容。

### 16.5 兼容与安全

- V1 包和既有会话升级后仍可读取和继续使用。
- 现有无项目智能体会话保持无项目状态，消息和固定版本不变。
- 同版本不同内容、发布者指纹变化、checksum 或签名错误继续被拒绝。
- 人物包、会话上下文、可选项目文件和关系记忆任一层缺失时都有明确降级，不导致普通会话回归。

### 16.6 工程验证

- 领域测试覆盖 4.2 中四种 `projectId/agentId` 组合，以及非法 `agentId/agentVersion` 组合。
- Room 11 -> 12 迁移保留既有会话、消息、Markdown 弱关联和智能体索引。
- UI 测试覆盖首页新建、项目内新建、安装后开始对话、首条消息后锁定身份和停用身份回退。
- 运行时测试覆盖 V1/V2 包、无项目人格会话、人格项目会话、项目删除和关系记忆清空。
- 李德胜 V1 示例问题加入回归集，验证正文不再出现资料编号、惯例免责声明和机械三段式。
