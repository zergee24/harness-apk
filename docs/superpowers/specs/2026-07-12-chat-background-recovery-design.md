# 会话后台执行恢复设计

## 问题

会话执行服务当前返回 `START_NOT_STICKY`。系统在应用位于后台时回收服务或进程后，不会重建服务；应用下次创建又将 `RUNNING` 执行直接标记为“已中断”，导致用户看到流式回复被切断。

## 决策

将前台服务改为 `START_REDELIVER_INTENT`，并让协调器在发现数据库中存在、但本进程没有 runner 的 `RUNNING` 执行时，将其恢复为 `QUEUED` 后从原用户消息重新发起请求。

- 已产生的助手部分回复标记为已取消并保留，恢复执行创建新的助手回复，避免覆盖已见内容。
- 同一会话仍保持单 runner FIFO；不同会话仍可并行。
- 服务仅在没有 `QUEUED` 或 `RUNNING` 执行时停止，避免启动与状态切换窗口错误停止前台服务。
- 服务不承诺在用户从系统设置强制停止应用后恢复；该情形由 Android 平台明确终止。
- `dataSync` 前台服务持续超过 Android 的时限时必须自行停止，不把超时作为普通会话中断处理。

## 验证

- 单元测试覆盖：服务回收后的运行中执行转回队列、已有本进程 runner 不被重复恢复、前台服务只有在无打开执行时停止。
- 在已连接设备上发送长回复，切换其他应用与锁屏后确认通知持续、回复完成且无“已中断”。
