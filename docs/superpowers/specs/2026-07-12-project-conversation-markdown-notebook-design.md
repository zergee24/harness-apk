# 项目会话驱动 Markdown 记事本设计

## 1. 目标与边界

Harness APK 的最基本记事本能力服务于项目内持续记录和整理。用户在一个项目会话中输入内容，系统自动把本轮对话组织为可审核的 Markdown 文件变更；用户确认后才写入项目文件。

本设计遵循以下第一性原则：

1. 项目是唯一工作边界。临时会话不进入记事本链路。
2. 会话与 Markdown 是并列实体，不互相拥有。
3. Markdown 正文只保存在项目文件系统；Room 不保存文件副本。
4. 所有文件写入必须在用户看过变更后明确应用。
5. 每条项目会话的文本发送自动规划 Markdown 变更，用户不需要输入“保存”或“写入文档”。

本期不新增独立笔记页、笔记目录、标签、模板市场、Markdown 专有编辑器、全文检索或跨项目笔记。

## 2. 当前基础与问题

当前项目已具备：

- `Conversation.projectId`：会话可归属项目。
- `FileProjectRepository`：项目内 Markdown 列表、读取、路径校验和受控写入。
- `MarkdownUpdatePlannerUseCase`：模型将文本转为 Markdown 更新计划。
- `MarkdownFileChangeController`：规划、审核、部分应用、失败重试和撤回状态。
- 项目页的“会话 / 文件夹 / Git”三个视图。

但当前链路存在三个问题：

1. 会话上下文默认索引项目全部 Markdown，项目变大后会引入无关内容和 token 成本。
2. 用户需要显式触发“文件变更模式”或“写回 Markdown”，正常对话与文档沉淀是两条交互。
3. 会话与文件之间没有可演进的关联；同一文件无法被多个会话有选择地复用。

## 3. 领域模型

```text
Project
  |- Conversation (0..N)
  |- Markdown file (0..N)

Conversation <-- weak many-to-many --> Markdown file
```

### 3.1 会话

会话继续是聊天历史、标题、模型选择和项目归属的载体。项目会话可以关联零到多份 Markdown；临时会话没有 Markdown 自动沉淀行为。

### 3.2 Markdown 文件

项目目录中的任意 `.md` 都可被会话关联。文件仍可通过项目文件夹预览、手工编辑、导出、外部应用打开和 Git 管理。文件不写入 Harness 专有 front matter 或会话标记。

### 3.3 弱关联

新增 `conversation_markdown_links` 表：

| 字段 | 说明 |
| --- | --- |
| `conversationId` | 会话 ID |
| `projectId` | 项目 ID，必须与会话归属一致 |
| `relativePath` | 项目内规范化的 Markdown 相对路径 |
| `linkedAt` | 首次关联时间 |
| `updatedAt` | 最近一次关联变动时间 |

唯一约束为 `(conversationId, projectId, relativePath)`。该表是弱引用而不是文件目录表：

- 不对文件建立数据库外键，也不从文件推导或写回关联信息。
- 文件被项目内手工编辑、Git 操作、项目包导入覆盖或删除后，关联可以失效。
- 失效关联保留在数据库中，但不会进入模型上下文；关联面板显示“文件已不可用”，支持移除或重新关联。
- 会话归档只隐藏关联，不删除 Markdown；移除全部关联也不删除文件。

## 4. 自动会话闭环

### 4.1 发送规则

当用户在 `projectId` 非空的会话中发送纯文本：

1. 持久化用户消息。
2. 读取该会话的有效关联文件；跳过缺失文件和非 `.md` 路径。
3. 以会话上下文、项目上下文、有效关联文件和本轮用户输入发起一次结构化模型请求。
4. 解析为助手回复和 `0..N` 个 Markdown 更新提案。
5. 持久化助手回复，并将提案以现有文件变更卡片展示。

临时会话和图片附件继续走现有普通聊天链路；本期自动沉淀只覆盖项目会话的纯文本输入，避免在第一版为多模态输入制造不可靠的文件写入。

### 4.2 结构化模型输出

自动请求一次完成“会话回复”和“文件变更规划”，不先发普通聊天再额外调用规划模型。响应结构为：

```json
{
  "reply": "本轮结论或下一步建议",
  "updates": [
    {
      "operation": "create",
      "path": "docs/weekly-review.md",
      "title": "本周复盘",
      "reason": "沉淀本轮确认的风险与下一步",
      "markdown": "# 本周复盘\\n"
    }
  ]
}
```

模型可返回空 `updates`，此时只展示助手回复，不显示“无变更”卡片。对已关联文件，系统提示模型仅在本轮确有影响时更新；模型不得因文件被关联而机械重写。模型可以提议创建项目内新的 `.md`，但仍必须经过审核和路径校验。

若结构化响应无法解析：保留原始模型文本为助手回复，不写入任何文件；同一轮用户消息旁提供“重新生成变更”，重新使用原消息和当前文件内容，不重复插入用户消息。

### 4.3 审核与应用

用户路径固定为：

```text
输入 -> 回复与变更摘要 -> 查看 diff -> 应用
```

“查看 diff”回答准备修改什么，“应用”授权真实写入，两者都保留。移除现有需要用户先选择“生成 Markdown 变更”或“写回 Markdown”的额外触发步骤。

写入使用现有逐文件结果模型：

- 全部成功：卡片显示已写入；成功的新增文件自动弱关联到当前会话。
- 部分成功：保留已写入结果，只重试失败项；仅成功的新文件创建关联。
- 全部失败：不创建新关联，保留失败原因和重试入口。

## 5. 并发与外部变更保护

多对多会话和其他项目写入路径意味着写入前不能假设文件仍保持规划时的内容。

每个更新提案保存其规划时的基线内容摘要：

- `update`：保存目标文件的 SHA-256 内容摘要。
- `create`：保存“规划时目标路径不存在”的基线状态。

用户点击应用时，Repository 重新读取每个目标：

- 基线一致，允许写入。
- `update` 的摘要变化，或 `create` 的路径已存在，拒绝覆盖该文件。
- 卡片显示“文件已更新”，用户可一键基于最新文件重新生成本轮变更。

该检查按文件执行。一个文件冲突不会阻断同一批其他无冲突文件的写入；结果仍通过部分成功卡片呈现。

## 6. 交互设计

### 6.1 项目页

不增加“笔记”页签。现有“会话”仍是项目工作的主入口，“文件夹”仍是实际文件视图。

- 新建项目会话自动带入当前 `projectId`。
- 文件夹中选中 Markdown 时，在现有预览操作旁提供“在会话中讨论”图标。点击后创建一个项目会话、建立该文件的弱关联并打开会话。
- 非 Markdown 文件不显示该图标。

### 6.2 会话页

会话顶部仅增加一个文件图标和有效关联数量，例如“文件 2”。没有常驻文件列表、笔记栏或第二套导航。

点击图标打开关联面板：

- 展示已关联文件及不可用状态。
- 支持搜索项目 Markdown、勾选多个文件并一次确认。
- 支持移除关联；移除不删除文件。
- 选择结果立即影响下一轮自动规划的模型上下文。

### 6.3 关联演进

一个文件可通过项目文件夹快速开始新会话，也可被任意已有项目会话从关联面板加入。会话和文件都可独立存在；关联只是可增减的协作线索。

## 7. 存储与权限原则

项目文件继续使用既有 `FileProjectRepository` 的 app-specific 项目目录（当前位于 `appContext.filesDir/projects`），本期不迁移存储位置、不申请广泛文件管理权限，也不为记事本创建第二份私有副本。Android 将内部 app-specific 存储定位为仅供应用访问的持久目录，但应用卸载会移除其中的文件；这一生命周期属于现有项目架构，本设计不改变它。[Android app-specific 文件存储](https://developer.android.com/training/data-storage/app-specific)

## 8. 错误处理

| 场景 | 行为 |
| --- | --- |
| 没有可用模型 | 复用现有模型配置错误，不创建文件变更 |
| 关联文件缺失 | 从本轮上下文排除；在关联面板可移除或重选 |
| 结构化响应解析失败 | 保留原始回复，不写文件，提供重新生成变更 |
| 路径非法或非 `.md` | 单项失败，不写入项目目录外文件 |
| 应用前文件已改变 | 单项拒绝覆盖，提示基于最新版本重新生成 |
| 部分写入失败 | 保留成功项与其关联，仅重试失败项 |
| 会话不属于目标项目 | 拒绝建立关联和写入，不隐式迁移会话 |

## 9. 实施范围

### 9.1 需要修改

- Room：新增弱关联实体、DAO、数据库迁移与 Repository API。
- 会话领域：读取、添加、移除、过滤有效关联；校验会话和文件属于同一项目。
- Markdown 规划：新增“回复 + 更新提案”的结构化响应模型和解析器。
- 文件变更：在提案和应用结果中增加基线摘要，执行写前冲突检查。
- 聊天 UI：项目纯文本发送进入自动规划；展示关联文件入口和自动变更卡片。
- 项目 UI：Markdown 文件预览增加“在会话中讨论”直达操作。

### 9.2 不修改

- 临时会话的普通聊天和现有图片输入链路。
- Git 提交、推送、项目导入导出、PDF 导出和文件手工编辑的产品边界。
- 项目文件格式；不注入 front matter、隐藏目录或元数据文件。

## 10. 测试与验收

单元测试至少覆盖：

1. 弱关联的新增、去重、移除、跨项目拒绝和失效路径过滤。
2. 自动请求只携带当前会话的有效关联 Markdown，不携带项目无关文件。
3. 结构化响应正确拆分为助手回复与零、一、多个更新提案。
4. 结构化解析失败时不产生写入提案，且可从原用户消息重新规划。
5. 成功创建的新文件自动关联；部分失败只关联成功的新文件。
6. 基线摘要变化时拒绝覆盖；同批其他文件可继续写入。
7. 项目纯文本会话走自动规划，临时会话与图片附件继续走普通聊天。
8. “在会话中讨论”正确创建项目会话并建立 Markdown 弱关联。

验收标准：

1. 用户可在一个项目会话中关联两份 Markdown，并让模型只对相关文件提出更新。
2. 同一 Markdown 可被两个项目会话关联，任一会话不会自动删除或独占该文件。
3. 用户不输入“保存”仍能在每次项目文本发送后看到变更提案。
4. 用户未点击应用时，项目目录和 Git 工作区不发生变化。
5. 审核期间文件被外部修改时，应用不会覆盖该修改。
6. 临时会话的聊天体验和现有项目文件操作不回归。
