# 异常 Markdown 代码围栏恢复设计

## 背景与根因

聊天回复中出现了整段 Markdown 被渲染为代码卡片、标题和粗体标记直接显示、长行横向溢出的问题。截图对应的模型输出包含两类不规范代码围栏：

- 关闭标记 ` ``` ` 紧贴在正文行尾，而不是独占一行。
- 未闭合的 `text` 围栏后继续输出分隔线、标题和普通正文。

当前 `normalizeModelMarkdown()` 仅在行首切换围栏状态。更关键的是，它把围栏内形如 ` ```text...` 的行当作关闭标记，但 CommonMark 只接受围栏字符后为空白的关闭行。归一化器因此认为代码块已经结束，而 CommonMark 仍把后续内容放在同一个代码块里，最终形成截图中的渲染错误。

## 目标

- 正确恢复模型输出中行尾紧贴的关闭围栏。
- 对明显进入 Markdown 正文结构的未闭合 `text` 代码块进行保守恢复。
- 标题、粗体、分隔线和正文重新按 Markdown 块渲染。
- 保持正常代码块长行横向滚动，不用自动换行掩盖解析错误。
- 不改写持久化消息原文，只在展示前的解析归一化阶段处理。

## 非目标

- 不尝试自动修复任意损坏的 Markdown。
- 不改变 Kotlin、Java、Shell 等真实代码块中的 `#`、`---` 或其他合法源码。
- 不处理截图中的 `Software caused connection abort` 网络原因；该错误继续独立显示并允许复制日志。
- 不重构消息存储、流式事件或整个 Markdown 渲染架构。

## 修复方案

归一化器显式记录当前围栏的字符、长度和 info，而不是对任意行首围栏简单取反。

### 1. 与 CommonMark 对齐关闭规则

只有满足以下条件的行才作为正常关闭围栏：

- 行首最多三个空格。
- 围栏字符与打开围栏一致。
- 围栏长度不短于打开围栏。
- 围栏字符后只允许空白。

这样 ` ```text左侧16:18:` 在围栏内部不会再被误认为合法关闭行。

### 2. 恢复行尾关闭围栏

当处于代码块内部，且非空内容行以与打开围栏匹配的 ` ``` ` 或 ` ~~~ ` 结束时，将该行拆为：

1. 围栏前的代码内容；
2. 独占一行的合法关闭围栏。

恢复只针对围栏结尾，保留结尾前的全部内容。

### 3. 保守恢复未闭合文本围栏

仅当打开围栏的 info 以 `text` 开头，并且已经出现空行时，遇到明确的 Markdown 正文边界才临时补关闭围栏：

- 分隔线；
- 标题；
- 引用开头。

此限制避免把真实编程语言代码中的 `#` 或 `---` 错误截断。截图中的第二个 `text` 块会在 `---` 前结束，后续 `##` 标题和 `> **...**` 引用恢复正常渲染。

## 测试设计

先添加失败的单元回归测试，再实现解析修复：

1. 截图语料：两个 `text` 块，首块使用行尾关闭，次块缺少关闭并在分隔线后进入标题和引用。
2. 断言生成两个代码块，且分隔线、标题、引用不属于代码 literal。
3. 断言标题和强调标记解析为结构化 Markdown，不泄漏 `##`、`>`、`**` 原始标记。
4. 增加保护测试：Kotlin 代码块内的 `---`、`#`、Markdown 表格文本不触发恢复。
5. 运行 Markdown 单元测试、完整 Debug 单元测试和 Debug 构建。
6. 有可用模拟器时安装并做截图/UI 树复验；当前未检测到已连接设备时，将明确报告这一限制。

### 异常模型输出语料矩阵

除截图语料外，测试至少覆盖以下组合：

| 类别 | 异常输入 | 期望 |
|---|---|---|
| 反引号围栏 | 内容行尾紧贴 ` ``` ` | 拆出合法关闭行，正文不丢失 |
| 波浪线围栏 | 内容行尾紧贴 ` ~~~ ` | 与反引号围栏一致恢复 |
| 围栏 info | 围栏内再次出现 ` ```text...` | 不误判为合法关闭行 |
| 未闭合文本块 | 空行后出现 `---`、标题或引用 | 在正文边界前结束文本块 |
| 未闭合真实代码 | Kotlin、Java、Shell 内出现 `#`、`---`、表格字符 | 不进行猜测性截断 |
| 紧凑标题 | `##标题###子标题` | 拆为对应层级标题，不泄漏 `#` |
| 引用与强调 | `> **正文**` | 保留引用和强调 AST |
| 表格 | 紧凑 GFM 表格紧随恢复后的代码块 | 解析成表格，不显示分隔标记 |
| 列表 | 有序、无序、任务和嵌套列表 | 列表层级和任务状态不丢失 |
| 数学块 | 未闭合公式后进入标题或列表 | 继续满足既有公式恢复行为 |
| 流式尾部 | 半截 fence、表格、公式逐步追加 | 稳定块不回退，尾部不吞掉前文 |
| Unicode 长文本 | 中文、英文、emoji 和超长无空格 token | 不崩溃、不丢字符，代码长行仍可横滚 |

### 渲染层验证

- Compose 测试断言恢复后的标题、正文和引用文本可见。
- Compose 测试断言原始 `##`、`|---|` 等结构标记不作为正文泄漏。
- 保留正常代码块的语言标签、复制按钮和横向滚动策略。
- 对截图宽度附近的窄屏做截图检查，确认普通正文换行且不被代码卡片裁切。

### 回归与随机组合检查

- 继续运行现有 `markdown` 回归语料目录中的全部样例。
- 新增一份损坏围栏回归语料，避免测试只依赖内联字符串。
- 用户已允许 mock 和生成长渲染文本。测试可生成超长代码块、宽表格、深层列表、混合中英文、emoji、公式、长 URL/无空格 token，以及故意损坏的围栏组合。
- 用固定种子的组合测试，把正文、围栏、标题、分隔线和列表按多种顺序拼接；固定种子保证失败可以稳定复现。基本不变量是解析不崩溃、输入中的可见正文字符不静默丢失。
- 生成语料仅用于本地单元测试、Compose 测试和模拟器 QA，不写入生产会话数据库，也不发往真实 LLM Provider。
- 对修复前后的 AST 做针对性比较，确认变化只发生在异常围栏语料，不影响正常语料。

## 验收标准

- 截图语料不再被解析成一个吞掉后文的超长代码块。
- 异常模型输出语料矩阵全部通过，且真实代码反例不会被提前截断。
- 正常代码围栏行为和既有 Markdown 回归语料全部通过。
- Compose 渲染测试证明结构标记不泄漏，窄屏正文不再因误判代码块而横向裁切。
- 应用 Debug 构建成功。
- 代码差异只涉及解析器、回归测试和本设计文档。

## 后续修正：避免 `text` 与中文正文粘连时误建代码块

### 回归原因

初版恢复规则把所有 info 以 `text` 开头的围栏视为文本代码块。CommonMark 本身允许任意 info，因此以下异常模型输出会被解析为代码块：

````text
```text一级标题
普通正文
```
````

其中 `一级标题` 实际是正文，却与 `text` 语言标记粘连。只把 `isTextLike` 从前缀匹配改成精确匹配仍不够，因为 CommonMark 依然会创建代码块。

### 选定方案

在 CommonMark 解析前增加一层保守解包：

- 标准 ` ```text`、` ```text ` 以及正常编程语言围栏继续作为代码块。
- 当 info 以 `text` 开头，且紧随其后的第一个字符属于中文正文时，判定为模型把语言标记和正文粘连。
- 对多行异常围栏：去掉打开/关闭标记，把 `text` 后的中文内容和围栏内文本重新交给普通 Markdown 解析。
- 对单行异常围栏 ` ```text一级内容``` `：去掉两端围栏和 `text` 前缀，按普通 Markdown 行解析。
- 不对 `plaintext`、`text/plain`、未知编程语言或只含 ASCII info 的围栏做猜测性解包。

### 新增测试

1. ` ```text一级标题 ... ``` ` 不产生 `MarkdownBlock.Code`，标题和正文字符不丢失。
2. ` ```text左侧布局``` ` 单行异常围栏按普通段落显示。
3. 标准 ` ```text` 日志仍产生代码块。
4. ` ```plaintext`、` ```text/plain` 和 Kotlin 代码块保持代码块。
5. 一整段混合 Markdown 中，只有明确的真实代码围栏产生代码块。
6. 重新运行长语料、固定种子组合、全部单元测试和模拟器 Compose 测试。
