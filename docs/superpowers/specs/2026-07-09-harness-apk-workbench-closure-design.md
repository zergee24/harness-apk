# Harness APK 工作台闭环详细规格

## 结论

本规格覆盖 Harness APK 当前路线图中保留的 P0、P1、P3：

- P0：稳定可用闭环。
- P1：项目工作台增强。
- P3：分发、更新和长期维护。

原 P2“模型与工具能力扩展”不进入本规格。Provider、联网搜索、语音、技能和插件能力只保留现状，后续只修复阻塞核心闭环的问题，不做独立扩展、不做能力矩阵、不新增插件执行平台。

产品主线是：用户在 Android 手机上进入项目，开启项目会话，让 LLM 生成文件变更，审核 diff 后写回项目文件，再查看文件和 Git 状态，最后通过稳定的 APK 更新链路持续获得新版本。

## 背景和问题

Harness APK 已经从“家用 LLM 聊天 APK”演进为移动端个人 AI 项目工作台。当前代码已经具备会话、项目、文件夹、Markdown 变更计划、Git 面板、项目导入导出、PDF 导出、自更新等能力，但产品边界还需要进一步收敛：

- 聊天和文件变更容易混在一起，用户不知道什么时候该普通发送，什么时候该生成文件变更。
- 项目工作台已有多个 tab，但“项目会话、文件夹、Git 状态”之间的主次关系还需要更稳定。
- Markdown 写回已有 diff 审核，但写回后的文件夹、预览和 Git 状态刷新要形成可靠闭环。
- Git 能力已经包含初始化、克隆、提交、推送、fetch、快进拉取和分支切换，但移动端必须限制在低风险动作内。
- 自更新链路已经支持 manifest、分片、SHA-256 和安装器，但发布前验收和失败处理还需要产品化。

## 产品对象

### 项目

项目是长期工作容器，对应本机一个项目目录。项目包含会话、上下文、文件交付物和可选 Git 仓库。

项目必须回答：

- 当前在做哪个项目。
- 项目里有哪些会话。
- 项目沉淀了哪些文件。
- 项目文件当前有哪些 Git 变更。

### 项目会话

项目会话是项目工作的主要入口，本质上仍是普通会话，只是通过 `projectId` 归属到项目。

项目会话承担：

- 获取项目上下文。
- 让 LLM 基于项目现有文件生成回复或文件变更。
- 产生可审核的 Markdown 更新。
- 在写回后记录系统事件。

项目会话不承担：

- 直接静默写文件。
- 自动提交 Git。
- 替代项目文件夹的完整文件管理。

### 文件变更草稿

文件变更草稿是一次 LLM 文件写作请求的中间结果。它不是最终项目文件，必须经过用户审核后才允许写入。

草稿包含：

- 关联会话。
- 关联项目。
- 来源用户消息。
- 变更状态。
- 变更文件列表。
- 每个文件的目标路径、标题、原因、完整 Markdown 和 diff 统计。

第一阶段草稿可以继续保存在会话 UI 内存状态；如果后续要求 App 重启后继续审核，再持久化到 Room。

### 项目文件

项目文件是项目目录下的真实产物。当前重点是 Markdown，文件夹视图也可索引 PDF、Office、代码、图片、文本和其他文件。

项目文件必须通过项目目录边界校验，不允许路径逃逸。LLM 生成链路只允许写入 Markdown 文件，不允许删除文件。

### Git 状态

Git 状态是项目文件变化的二次确认层。它用于回答：

- 当前项目是否是 Git 仓库。
- 当前分支是什么。
- 是否有本地改动。
- 本地相对远端领先或落后多少。
- 哪些文件发生了变化。

Git 操作必须保持移动端低风险：不自动解决冲突，不强制推送，不在脏工作区切分支或拉取。

### 更新包

更新包由静态 manifest、APK 完整包或分片、SHA-256 和发布说明组成。

客户端只信任 HTTPS 下载地址、APK 签名和 SHA-256 校验。更新失败不能破坏当前已安装版本。

## 范围

### P0 范围：稳定可用闭环

P0 要确保一个真实设备用户能完成：

1. 创建或导入项目。
2. 进入项目会话。
3. 选择“生成文件变更”。
4. 等待 LLM 生成 Markdown 更新计划。
5. 查看文件变更卡片。
6. 打开 diff 审核。
7. 保留或撤回文件级变更。
8. 应用保留项。
9. 在文件夹看到写入结果。
10. 在 Git tab 看到项目变更。
11. 通过更新页检查、下载、校验和安装新版本。

### P1 范围：项目工作台增强

P1 让项目页成为长期工作的默认入口：

- 项目切换、创建、删除、导入、导出、分享入口清晰。
- 项目默认展示会话 tab。
- 文件夹 tab 支持搜索、筛选、目录树、预览、编辑、保存、打开、分享和 PDF 导出。
- Git tab 支持状态查看、初始化、克隆、fetch、快进拉取、提交、推送、新建分支和切换分支。
- 项目上下文作为会话输入的一部分，可被读取和维护。

### P3 范围：分发、更新和长期维护

P3 固化持续发布和维护：

- 测试通道和正式通道分离。
- Debug 包只读取测试通道 manifest。
- Release 包只读取正式通道 manifest。
- manifest、APK 和分片必须 HTTPS。
- 客户端优先使用分片下载，校验合格后拉起系统安装器。
- 正式包必须签名后上传。
- 发布前有固定回归清单。

## 不做范围

- 不做原 P2 的模型能力矩阵。
- 不新增 Provider 能力配置体系。
- 不扩展联网搜索路由。
- 不扩展语音助手能力。
- 不开放第三方插件执行。
- 不做账号、云同步、多人协作和家庭后台。
- 不做应用商店发布、支付、订阅和额度管理。
- 不做静默安装或后台无感升级。
- 不做自动提交、自动推送或无审核写文件。
- 不做完整 IDE 式代码编辑器。
- 不在移动端处理复杂 Git 冲突。

## 主链路

### 主链路 1：项目文件变更闭环

入口：

- 项目页点击“新建项目会话”。
- 一级会话页打开已有项目会话。
- 聊天页选择项目后发送文件变更请求。

流程：

1. 用户进入一个带 `projectId` 的会话。
2. 聊天页加载项目列表、当前项目、项目上下文和项目 Markdown 快照。
3. 用户在输入框填写文件写作请求。
4. 用户点击“生成文件变更”。
5. 系统校验当前会话必须已选择项目、输入不能为空、文件变更模式不带图片、助手不能正在生成。
6. 系统插入用户消息。
7. 系统创建 `PLANNING` 文件变更草稿。
8. `MarkdownUpdatePlannerUseCase` 基于项目名称、项目上下文、现有 Markdown 和用户请求生成结构化更新计划。
9. 系统解析更新计划并计算每个文件的 diff 和新增/删除行数。
10. 草稿进入 `READY`，聊天流展示文件变更卡片。
11. 用户可以查看 diff、应用保留项、撤回或失败后重试。
12. 用户确认应用后，系统通过 `ProjectWorkspaceGateway.applyMarkdownUpdates` 写入项目。
13. 系统插入“已沉淀到项目”事件，列出实际写入路径。
14. 系统刷新当前项目文件列表。
15. 项目页文件夹和 Git tab 可看到结果。

阻断规则：

- 没有项目：阻断并提示“请先选择项目”。
- 输入为空：不发送。
- 带图片：阻断并提示“文件变更模式暂不支持图片输入”。
- 助手忙碌：不重复发送。
- LLM 未返回可解析更新计划：草稿进入 `FAILED`。
- 更新计划为空：草稿进入 `FAILED`。
- 所有文件都被撤回：草稿进入 `DISMISSED`，不写文件。

### 主链路 2：项目工作台

入口：

- 首页模式切换到“项目”。

流程：

1. 项目页加载本机项目列表。
2. 默认选择最近更新项目。
3. 顶部展示当前项目、项目切换、新建项目、克隆仓库、导入、导出、分享、删除和新建项目会话入口。
4. 默认 tab 是“会话”，展示当前项目关联会话。
5. 用户可切换到“文件夹”，查看项目文件树。
6. 用户可搜索、筛选、选择文件并预览。
7. 对可文本预览文件，用户可编辑并保存。
8. 对 Markdown 文件，用户可切换预览并导出 PDF。
9. 用户可外部打开或分享文件内容。
10. 用户可切换到“Git”，查看当前仓库状态和低风险 Git 操作。

页面原则：

- 项目会话是默认工作入口。
- 文件夹是沉淀结果视图，不作为 LLM 写入的绕过入口。
- Git 是变更确认和同步入口，不替代桌面端处理复杂冲突。

### 主链路 3：低风险 Git 操作

入口：

- 项目页 Git tab。
- 设置页 Git / Gitee 配置。

流程：

1. 用户在设置页保存 Gitee 用户名、私人令牌、作者名和作者邮箱。
2. 项目 Git tab 判断当前项目是否是 Git 仓库。
3. 如果不是仓库，用户可初始化本地 Git 或克隆远端仓库为新项目。
4. 如果是仓库，展示当前分支、干净/有改动、领先/落后、本地变更文件。
5. 用户可刷新状态。
6. 用户可 fetch 远端信息。
7. 用户可在工作区干净时执行快进拉取。
8. 用户可在工作区干净时新建并切换分支，或切换已有分支。
9. 用户可提交全部当前改动。
10. 用户可推送当前分支到 origin。

低风险规则：

- 切换分支前必须工作区干净。
- 快进拉取前必须工作区干净，只允许 fast-forward。
- 非 fast-forward、冲突和非安全仓库状态必须阻断，提示到桌面端处理。
- 不允许 force push。
- 不自动提交。
- 提交前必须展示提交确认，不允许空提交信息。
- 推送失败必须保留本地提交。

P1 需要增强：

- 提交确认弹窗应展示本次将提交的文件统计和文件列表。
- 推送前应展示当前分支、领先/落后状态和远端名称。
- 当前项目没有 origin 时，推送入口应给出可理解错误或后续绑定远端入口。
- 冲突文件应在变更列表中突出显示，并禁用拉取、切换和提交。

### 主链路 4：APK 更新和发布维护

入口：

- 设置页“检查更新”。
- App 启动后的更新检查入口。
- 发布侧 GitHub Actions 或本地打包脚本。

客户端流程：

1. 客户端读取当前版本号和版本码。
2. 请求对应通道的 HTTPS manifest。
3. 解析 `versionCode`、`versionName`、`minSupportedVersionCode`、`apkUrl`、`apkChunks`、`sha256`、`releaseNotes` 和 `publishedAt`。
4. 如果当前版本小于 `minSupportedVersionCode`，标记强制更新。
5. 如果 manifest 版本高于当前版本，标记发现新版本。
6. 客户端下载 APK 分片；没有分片时下载完整包。
7. 下载完成后计算 SHA-256。
8. 校验失败则删除下载文件。
9. 校验成功后拉起系统安装器。
10. 如果缺少“安装未知应用”权限，跳转系统授权页，返回后继续安装。

发布流程：

1. 测试通道构建 debug 包。
2. 正式通道构建 release 包。
3. 正式通道必须具备签名 secrets。
4. 生成 update.json、APK 和 chunks。
5. 上传到 OSS 对应前缀。
6. 不把 APK、分片和生产 update.json 提交到源码仓库。

失败规则：

- manifest 请求失败：提示更新检查失败，允许重试。
- manifest 缺字段或字段非法：提示更新清单不可用。
- 下载失败：删除不完整文件，允许重试。
- SHA-256 不匹配：删除下载文件，提示安装包校验失败。
- 用户取消安装：不重复打扰，更新页保留继续安装入口。
- 强制更新失败：允许重试或退出，不破坏当前安装包。

## 状态机

### 文件变更草稿状态

`PLANNING`：

- 含义：正在让 LLM 生成 Markdown 更新计划。
- 展示：文件变更卡片显示生成中。
- 可操作：无，允许全局停止后进入失败或保留当前状态。

`READY`：

- 含义：已生成可审核文件变更。
- 展示：文件列表、操作类型、标题、原因、增加/删除行数。
- 可操作：查看 diff、应用全部/保留项、撤回。

`APPLIED`：

- 含义：保留项已写入项目。
- 展示：已应用数量和写入路径事件。
- 可操作：只读查看，不重复应用。

`DISMISSED`：

- 含义：用户撤回变更或保留项为空。
- 展示：已撤回。
- 可操作：只读查看；如果要重做，需要重新发起文件变更请求。

`FAILED`：

- 含义：生成、解析或写入失败。
- 展示：失败原因。
- 可操作：重试、撤回。

### 项目状态

`EMPTY`：

- 没有项目。
- 展示空状态，引导创建项目或克隆仓库。

`PROJECT_SELECTED`：

- 已选择项目。
- 默认展示会话 tab。

`PROJECT_LOADING_FILES`：

- 正在加载或搜索文件。
- 文件夹 tab 可显示加载态，避免旧数据误导。

`PROJECT_READY`：

- 会话、文件夹和 Git 状态均可切换查看。

`PROJECT_ERROR`：

- 项目读取、导入、导出或删除失败。
- 以可关闭状态 chip 或错误卡片展示原因。

### Git 状态

`NOT_REPOSITORY`：

- 项目不是 Git 仓库。
- 可初始化或克隆。

`CLEAN`：

- 工作区干净。
- 允许 fetch、快进拉取、切换分支、新建分支和推送。

`DIRTY`：

- 有本地改动。
- 允许提交、刷新、fetch 和推送已有提交。
- 禁用快进拉取和切换分支。

`AHEAD`：

- 当前分支领先远端。
- 允许推送。

`BEHIND`：

- 当前分支落后远端。
- 工作区干净时允许快进拉取。

`CONFLICTING`：

- 存在冲突。
- 禁用提交、推送、拉取和切换，提示到桌面端处理。

`AUTH_MISSING_OR_FAILED`：

- 认证缺失或失败。
- 引导到 Git / Gitee 设置页。

### 更新状态

`IDLE`：未检查。

`CHECKING`：正在请求 manifest。

`LATEST`：当前已是最新。

`UPDATE_AVAILABLE`：发现新版本。

`FORCE_UPDATE`：当前版本低于最低支持版本。

`DOWNLOADING`：正在下载完整包或分片。

`READY_TO_INSTALL`：下载并校验通过。

`PERMISSION_REQUIRED`：需要授权安装未知应用。

`INSTALLER_OPENED`：已拉起系统安装器。

`FAILED`：检查、下载、校验或安装启动失败。

## 页面与入口

### 首页

首页保留“会话 / 项目”模式切换。

会话模式：

- 展示最近会话。
- 保留快速新建普通会话。
- 项目会话可出现在最近会话中，但需要显示项目归属。

项目模式：

- 展示项目工作台。
- 顶部标题跟随当前项目。

### 项目页

项目页包含：

- 项目 header。
- 状态提示。
- 二级 tab：会话、文件夹、Git。

项目 header 必须提供：

- 当前项目名称。
- 切换项目。
- 新建项目。
- 克隆仓库。
- 新建项目会话。
- 导入项目包。
- 导出项目包。
- 分享项目包。
- 删除项目。

删除项目必须二次确认，并明确说明删除本机项目文件，历史会话保留。

### 会话 tab

展示当前项目下的会话，按更新时间倒序。

空状态：

- 提示用户新建项目会话。

会话项：

- 展示标题。
- 展示最近更新时间。
- 如果有提示词配置，展示“已设提示词”状态。

### 文件夹 tab

包含：

- 搜索框。
- 类型筛选：全部、Markdown、Office、PDF、代码、图片、其他。
- 目录树。
- 预览/编辑面板。

预览规则：

- Markdown：支持 Markdown 预览、编辑、保存、PDF 导出。
- 文本和代码：支持文本预览、编辑、保存。
- PDF/Office/图片：不在 App 内完整解析，提供外部打开。
- 大文件：后续可增加大小限制和提示；本规格不要求复杂分页。

### Git tab

包含：

- 仓库状态卡。
- 当前分支和 clean/dirty 状态。
- ahead/behind 状态。
- 操作按钮。
- 分支列表。
- 变更文件列表。

操作按钮的显示规则：

- 非 Git 仓库：显示初始化 Git、克隆仓库。
- Git 仓库：显示提交、推送、Fetch、快进拉取、新建分支、刷新。
- 工作区不干净时：禁用快进拉取和分支切换。
- 有冲突时：禁用高风险动作。

### 聊天页

聊天页保留两类发送：

- 普通发送：进入 LLM 聊天流。
- 生成文件变更：进入 Markdown 文件变更草稿状态机。

文件变更入口要求：

- 当前会话必须选择项目。
- 输入不能为空。
- 不支持图片。
- 助手生成中不能重复触发。

聊天流展示：

- 普通助手消息继续流式渲染。
- 长 Markdown 写作应优先用文件变更卡片，不把完整长文档塞入聊天气泡。
- 写回成功后插入系统事件。

### 设置页

设置页保留现有入口：

- 模型配置。
- 搜索能力。
- 语音能力。
- Git / Gitee。
- 技能 / 插件。
- 检查更新。

本规格只要求 Git / Gitee 和检查更新服务于 P0/P1/P3。模型、搜索、语音、技能不做新增扩展。

## 字段与来源

### Conversation

关键字段：

- `id`：会话 ID。
- `title`：会话标题。
- `updatedAt`：更新时间。
- `projectId`：项目归属，可空。
- `promptOriginal`、`promptOptimized`、`promptFinal`：会话提示词相关字段。

来源：

- Room 会话表。

规则：

- 项目会话必须持久化 `projectId`。
- 普通会话 `projectId` 为空。
- 项目页会话 tab 只按 `projectId` 过滤，不通过标题推断。

### Project

关键字段：

- `id`：项目目录 ID。
- `name`：项目名称。
- `rootDirectory`：本机项目目录。
- `updatedAt`：更新时间。

来源：

- 本机项目目录。
- README 标题或本地 harness 元数据。

规则：

- 创建项目时生成 README、context.md 和默认目录结构。
- 导入项目时要校验 zip entry 路径，避免路径逃逸。

### ProjectDeliverable

关键字段：

- `id`：项目内相对路径。
- `title`：展示标题。
- `relativePath`：项目内相对路径。
- `template`：来源模板。
- `artifactType`：文件类型。
- `updatedAt`：文件更新时间。

来源：

- 项目目录文件扫描。

规则：

- 文件夹 tab 的展示基于真实项目文件。
- LLM 写入只允许 Markdown。
- PDF 导出会生成新的 PDF 文件并刷新文件列表。

### MarkdownFileChangeDraft

关键字段：

- `id`：草稿 ID。
- `conversationId`：所属会话。
- `projectId`：所属项目。
- `sourceUserMessageId`：来源用户消息。
- `status`：草稿状态。
- `summary`：展示摘要。
- `createdAt`、`updatedAt`：时间戳。

来源：

- 第一阶段为 UI 内存状态。

规则：

- `READY` 前不能应用。
- `APPLIED` 后不能重复应用。
- `DISMISSED` 后不能写入项目。
- `FAILED` 可以基于原始用户消息重试。

### MarkdownFileChangeItem

关键字段：

- `operation`：create 或 update。
- `path`：目标 Markdown 路径。
- `title`：文件标题。
- `reason`：生成或修改原因。
- `markdown`：完整 Markdown 内容。
- `addedLineCount`、`removedLineCount`：diff 统计。
- `retained`：是否保留。

规则：

- 路径必须是项目内相对路径。
- 路径必须以 `.md` 结尾。
- 不支持删除文件。
- 保留状态只影响本次应用，不直接改项目文件。

### GitStatusSummary

关键字段：

- `currentBranch`。
- `isClean`。
- `stagedCount`。
- `unstagedCount`。
- `untrackedCount`。
- `aheadCount`。
- `behindCount`。
- `files`。

来源：

- JGit。

规则：

- 状态读取失败不影响项目文件浏览。
- 认证失败引导到设置页。
- 冲突状态必须阻断高风险操作。

### UpdateManifest

关键字段：

- `versionCode`。
- `versionName`。
- `minSupportedVersionCode`。
- `apkUrl`。
- `apkChunks`。
- `sha256`。
- `releaseNotes`。
- `publishedAt`。

规则：

- `apkChunks` 非空时优先使用分片。
- `apkUrl` 和 `apkChunks` 至少有一个。
- 所有下载地址必须 HTTPS。
- SHA-256 必须匹配才允许安装。

## 权限和安全

### 本机数据

- API Key 继续本机加密保存。
- 项目文件保存在 App 管理的项目目录中。
- 项目导出包不应包含 API Key。
- 项目删除只删除项目文件，不删除历史会话。

### 文件写入

- 所有 LLM 写入必须经过 diff 审核。
- 项目路径必须经过 canonical path 边界校验。
- 不允许写入项目目录外。
- 不允许删除项目文件。
- 不允许静默覆盖未展示的文件。

### Git

- Token 只用于认证，不在 UI 明文回显。
- 提交身份由用户在设置中配置。
- 不允许 force push。
- 不自动提交。
- 拉取只允许 fast-forward。

### 更新

- 更新地址必须 HTTPS。
- APK 必须通过 SHA-256 校验。
- 正式包必须签名。
- 下载失败或校验失败必须删除不完整文件。

## 错误处理

### 文件变更错误

- 无项目：提示“请先选择项目”。
- 空输入：不发送，不产生草稿。
- 图片输入：提示“文件变更模式暂不支持图片输入”。
- LLM 非 JSON：展示“LLM 未返回 Markdown 更新 JSON”或解析错误。
- 空更新：展示“没有生成可审核的 Markdown 更新”。
- 写入失败：保留草稿，不清空 diff，允许重试或撤回。

### 项目错误

- 创建项目名称为空：提示“项目名称不能为空”。
- 导入包不可读：提示无法读取项目包。
- 导入路径不安全：提示不安全路径。
- 导出失败：提示无法写入项目包。
- 删除失败：提示删除项目失败。

### Git 错误

- 未配置认证：提示进入 Git / Gitee 设置。
- 工作区不干净时拉取或切分支：提示先提交或清理。
- 非快进拉取：提示到桌面端处理冲突后再同步。
- 推送失败：展示失败原因，保留本地提交。
- 仓库状态不安全：阻断操作。

### 更新错误

- HTTP 非成功：展示 HTTP 状态。
- manifest 缺 APK 地址：提示更新清单缺少 APK 下载地址。
- HTTP 下载失败：提示完整包或分片下载失败。
- SHA-256 不匹配：提示安装包校验失败。
- 无安装未知应用权限：引导到系统授权页。

## 测试策略

### 单元测试

文件变更：

- `decideFileChangeSend` 覆盖无项目、空输入、图片输入、助手忙碌和正常发送。
- `MarkdownFileChangeController` 覆盖 PLANNING、READY、FAILED、DISMISSED、APPLIED 状态转换。
- diff 统计覆盖新增、修改和空更新。
- retainedProposals 覆盖全保留、部分撤回和全撤回。

项目：

- 项目导入 zip 路径安全。
- 项目文件类型识别。
- 文件树构建和折叠。
- 搜索和筛选组合。
- PDF 导出路径生成。

Git：

- status 汇总 staged、unstaged、untracked、conflicting。
- 工作区不干净时禁止 checkout 和 fast-forward pull。
- 空提交信息阻断。
- 非 fast-forward pull 转换为用户可理解错误。

更新：

- manifest 解析。
- HTTPS 校验。
- apkUrl 和 apkChunks 选择。
- SHA-256 校验成功和失败。
- 下载失败删除临时文件。

### UI 状态测试

- 项目页默认进入会话 tab。
- 无项目空状态展示创建入口。
- 文件夹筛选和选择文件后预览状态正确。
- Git tab 非仓库状态显示初始化和克隆入口。
- 文件变更卡片各状态文案和按钮正确。
- 更新页检查中、发现更新、下载完成、失败状态展示正确。

### 人工验收

P0 验收：

1. 在真实 Android 设备安装测试 APK。
2. 配置一个可用 Provider。
3. 创建一个项目。
4. 新建项目会话。
5. 发送普通消息并获得回复。
6. 发送“生成一份项目计划 Markdown”文件变更请求。
7. 查看文件变更卡片。
8. 打开 diff，撤回其中一个文件或保留全部。
9. 应用变更。
10. 在文件夹 tab 查看新 Markdown。
11. 在 Git tab 查看变更文件。
12. 触发更新检查并完成测试通道 APK 下载校验。

P1 验收：

1. 导入一个项目 zip。
2. 搜索并筛选 Markdown、PDF、代码文件。
3. 编辑并保存一个文本文件。
4. 将 Markdown 导出 PDF。
5. 初始化 Git 仓库。
6. 创建分支。
7. 提交全部变更。
8. fetch 远端。
9. 工作区干净时快进拉取。
10. 工作区有改动时确认切分支和拉取被阻断。

P3 验收：

1. 手动运行测试通道发布。
2. 校验 update.json 包含完整包和分片。
3. 在客户端下载分片并合并。
4. 故意配置错误 SHA-256，确认客户端删除下载文件并提示校验失败。
5. 正式通道缺少签名 secrets 时确认不上传 unsigned APK。
6. 正式包签名配置完整时确认 release 包可下载、校验并拉起安装器。

## 验收标准

本规格完成时必须满足：

- 用户能在项目会话中明确区分普通聊天和文件变更。
- 文件变更必须经过 diff 审核才能写入项目。
- 写回成功后，聊天事件、文件夹和 Git 状态保持一致。
- 项目页默认支持长期项目工作，而不是只作为文件列表。
- Git 操作默认低风险，不处理复杂冲突，不强推，不自动提交。
- 测试通道 APK 可以稳定发布、检查、下载、校验和安装。
- 正式通道有签名和发布前验收保护。
- 原 P2 范围没有被引入实施计划。
