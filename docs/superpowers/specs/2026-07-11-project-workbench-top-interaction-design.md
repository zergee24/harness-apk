# 项目工作台顶部交互设计

## 目标

让用户进入项目工作台后，先清楚知道当前项目、可以立即做什么，以及会话、文件和 Git 分别处于什么状态。

本轮采用已确认的 A 方案：项目是工作台，内容继续通过标签页访问；不新增“任务”实体或第三层导航。

## 问题

当前顶部同时承载项目名称、项目切换、新建、导入导出和标签页，用户难以判断当前项目和下一步操作。标签页的内容能力已存在，但顶部没有明确表达它们的作用和当前状态。

## 信息结构

工作台仍保持两层：

1. 顶部项目区：当前项目及其主要操作。
2. 内容标签页：会话、文件夹、Git。

不改变 `ProjectWorkbenchTab`、项目选择、会话归属、文件交付物或 Git 数据流。

## 顶部项目区

### 当前项目行

- 当前项目名称是最高层级文字，可点开项目切换菜单。
- 旁边使用下拉图标表达可切换，不使用难以理解的卡片或项目编号。
- 无项目时明确显示“请选择项目”，并把“新建项目”作为主要操作。
- 克隆、导入、导出、分享、删除继续置于溢出菜单；删除保留确认对话框。

### 直接操作行

- 已选项目时，“新建项目会话”是唯一主按钮。
- 按钮附近显示三项简短状态：项目会话数量、文件数量、Git 工作区摘要。
- 状态不是可点击的第二套导航；点击标签页仍是进入详细内容的唯一入口。
- Git 未初始化或状态尚未加载时，显示清晰文字而非空白计数。

## 标签页

- 保留“会话 / 文件夹 / Git”的顺序和现有跳转目标。
- 标签采用固定高度 Material 3 样式；当前标签同时用文字强调和明确指示器表达。
- 标签下方增加一句与当前标签对应的短说明，仅在内容为空或首次展示时出现：
  - 会话：在当前项目内开始或继续工作。
  - 文件夹：查看会话沉淀和已写入文件。
  - Git：查看当前分支和工作区变更。
- 不在每个页面重复展示功能教学或快捷键说明。

## 交互与状态

- 切换项目后，顶部名称、状态摘要、会话列表、文件和 Git 状态都随现有选择刷新逻辑更新。
- 从聊天页跳转到文件夹或 Git 时，保留现有一次性 `ProjectWorkbenchTarget` 消费和刷新代际保护。
- Git 状态只在进入 Git 标签或已存在的安全刷新点加载；顶部摘要不得新增后台轮询。
- 异常或加载状态就近显示，不能让旧项目的状态短暂覆盖新项目。

## 实现边界

- 主要修改 `ProjectScreen.kt` 和 `ProjectUiState.kt`，必要时补充 `ProjectSessionLaunchUiStateTest.kt` 与 Compose 设备测试。
- 复用既有 `ProjectHeaderActionLayout`、`projectConversations`、交付物列表和 Git 状态对象。
- 不改数据库或 Repository API，不改聊天页、文件树和 Git 操作的业务规则。
- 不使用新的图片、网络依赖或设计系统。

## 验收

1. 有项目时，顶部先显示当前项目和“新建项目会话”。
2. 用户可从项目名称附近完成项目切换；低频操作不与主要操作并列。
3. 顶部能读到会话、文件和 Git 的简短状态，但不会创建第二套内容入口。
4. 三个标签仍可访问，聊天页的文件与 Git 跳转保持正确。
5. 无项目、加载、空项目、非 Git 项目和 Git 错误均有明确文字反馈。
6. 现有 JVM 测试、Debug 构建和相关 Compose 测试通过；设备可用时核对浅色、深色与放大字体下的顶部布局。

## 非目标

- 不建立项目内任务列表、任务状态、任务归档或跨项目任务移动。
- 不修改工作台的会话、文件夹和 Git 标签语义。
- 不新增项目成员、同步、远端仓库管理或自动提交。
