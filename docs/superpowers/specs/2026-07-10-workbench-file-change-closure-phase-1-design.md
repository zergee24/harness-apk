# Harness APK 工作台第一阶段：文件变更闭环设计

## 结论

工作台闭环按用户结果分四阶段实施，本设计只覆盖第一阶段：用户在项目会话中生成 Markdown 文件变更，审核后写入项目，并从聊天页直接查看真实文件和 Git 变更。

本阶段复用现有文件变更状态机、Markdown 更新计划、diff 审核和项目文件 Repository，不重做项目工作台，不扩展 Provider、搜索、语音或技能能力。

## 与总规格的关系

本设计是 `2026-07-09-harness-apk-workbench-closure-design.md` 中“实施包 1：文件变更闭环稳定”的可实施子规格。总规格继续描述 P0、P1、P3 的完整方向，实际交付按以下顺序推进：

1. 文件变更闭环。
2. 项目工作台闭环。
3. 低风险 Git 操作。
4. 更新与发布维护。

每一阶段独立设计、实施、验收和提交，不把后续阶段作为当前阶段完成的前置条件。

## 目标

第一阶段必须让用户稳定完成：

1. 在项目会话输入文件写作请求。
2. 生成一个或多个 Markdown 文件变更。
3. 审核 diff，并按文件保留或撤回。
4. 应用保留项。
5. 在聊天页看到真实写入结果。
6. 直接进入当前项目文件夹查看写入文件。
7. 直接进入当前项目 Git 标签页查看工作区变更。
8. 部分文件写入失败时，保留成功文件并只重试失败项。

## 非目标

- 不持久化文件变更草稿；App 进程终止后，未完成草稿可以丢失。
- 不支持 Markdown 以外的 LLM 文件写入。
- 不允许 LLM 删除项目文件。
- 不自动提交或推送 Git。
- 不重做项目工作台信息架构。
- 不扩展项目文件编辑器能力。
- 不新增 Provider、联网搜索、语音、技能或插件能力。
- 不把第二至第四阶段内容提前纳入本阶段。

## 用户流程

### 生成与审核

1. 用户进入带 `projectId` 的会话。
2. 用户输入文件写作请求并点击“生成文件变更”。
3. 系统校验项目、输入、图片和助手生成状态。
4. 系统插入用户消息并创建内存草稿。
5. `MarkdownUpdatePlannerUseCase` 生成结构化 Markdown 更新计划。
6. 系统解析计划、计算 diff 和增删行数。
7. 卡片进入 `READY`，用户打开审核弹窗。
8. 用户按文件保留或撤回，然后确认应用。

### 应用成功

1. 系统逐项写入保留文件，并收集每个文件的应用结果。
2. 所有文件成功时，草稿进入 `APPLIED`。
3. 聊天页显示实际写入数量和路径。
4. 用户可以继续聊天，也可以点击“查看文件”或“查看 Git 变更”。
5. “查看文件”进入当前项目文件夹，刷新文件列表并选中第一个写入文件。
6. “查看 Git 变更”进入当前项目 Git 标签页并刷新仓库状态。

### 部分成功

1. 已成功写入的文件保留在项目目录，不回滚。
2. 草稿进入 `PARTIALLY_APPLIED`。
3. 卡片分别展示成功数量、成功路径、失败数量、失败路径和原因。
4. “查看文件”和“查看 Git 变更”继续可用。
5. 用户可以点击“仅重试失败项”。
6. 重试只提交失败项，不重复写入已成功文件。
7. 重试后按累计结果重新判断：全部目标成功时进入 `APPLIED`；仍有成功项和失败项时保持 `PARTIALLY_APPLIED`；只有从未产生任何成功项时才进入 `FAILED`。

## 状态模型

### MarkdownFileChangeStatus

保留现有状态，并新增 `PARTIALLY_APPLIED`：

- `PLANNING`：正在生成更新计划。
- `READY`：已生成，等待审核或应用。
- `APPLIED`：所有保留项均写入成功。
- `PARTIALLY_APPLIED`：至少一个文件成功且至少一个文件失败。
- `DISMISSED`：用户撤回或没有保留项。
- `FAILED`：生成失败、解析失败，或本次应用没有任何文件成功。

### 单文件应用结果

新增结构化结果，不用异常代表整批结果：

```text
MarkdownFileApplyResult
- proposal
- status: SUCCEEDED | FAILED
- writtenDeliverable: CreatedDeliverable?
- errorMessage: String?
```

批量结果：

```text
MarkdownBatchApplyResult
- results
- succeeded
- failed
- isFullyApplied
- isPartiallyApplied
```

失败信息必须是可展示的脱敏消息，不包含 API Key、完整 Prompt 或敏感文件内容。

### 草稿展示状态

`MarkdownFileChangeState` 记录：

- 原始变更项和 diff。
- 实际成功写入路径。
- 失败项及错误信息。
- 当前仍可重试的失败项。

这些信息只在当前聊天页面生命周期内保存，不进入 Room。

## 写入策略

### 预校验

开始写入前，对所有保留项执行：

- 项目存在校验。
- 相对路径非空。
- `.md` 后缀校验。
- canonical path 项目目录边界校验。
- 不允许删除操作。

预校验失败的项目直接形成失败结果，不进入文件写入。

### 逐项执行

通过预校验的项目逐项执行写入，每一项独立捕获结果。单项失败不能中止后续项，也不能丢弃之前的成功结果。

本阶段不实现临时目录、备份和整批回滚。原因是当前项目文件写入本来就是逐文件执行，保留已完成成果比伪装成整批原子操作更符合真实状态，也能由 Git 工作区继续确认。

### 重试

“仅重试失败项”使用原始 proposal 和当前项目状态重新执行预校验与写入。成功项不参与重试。重试完成后合并新旧结果，确保成功路径不会丢失或重复计算。

## 跨页面导航

### ProjectWorkbenchTarget

新增应用级一次性工作台目标：

```text
ProjectWorkbenchTarget
- projectId
- tab: FILES | GIT
- selectedPath: String?
- requestKey
```

### 职责边界

- `ChatScreen`：展示应用结果并发出打开项目视图的意图。
- `HarnessApkApp`：保存一次性目标，切换到项目模式并返回首页。
- `ProjectScreen`：消费目标、选择项目和标签页、刷新并展示结果。
- `FileProjectRepository`：继续作为真实项目文件来源。
- `JGitEngine`：继续作为 Git 状态来源。

聊天页不直接调用项目页内部刷新函数，项目页也不读取聊天页内存状态。

### 查看文件

1. 目标包含当前 `projectId`、`FILES` 和第一个成功写入路径。
2. 项目页选中目标项目和文件夹标签页。
3. 项目页重新扫描文件。
4. 如果目标路径存在，选中并加载预览。
5. 如果目标路径暂未扫描到，显示文件夹列表并提示“文件已写入，请刷新后查看”。

### 查看 Git 变更

1. 目标包含当前 `projectId` 和 `GIT`。
2. 项目页选中目标项目和 Git 标签页。
3. 项目页重新读取仓库状态。
4. 非 Git 项目继续展示现有“初始化 Git”入口。

一次性目标消费后清空，避免重组、旋转屏幕或返回页面时重复跳转。

## 界面状态

### PLANNING

- 显示“正在生成文件变更”和进度状态。
- 不提供重复生成操作。

### READY

- 显示文件数量、文件名、操作类型和增删行数。
- 主操作为“审核变更”。
- 次操作为“撤回”。

### FAILED

- 使用错误语义色展示具体原因。
- 生成或解析失败时提供“重试”和“撤回”。
- 应用全部失败时保留 diff 和审核结果，允许重试失败项。

### DISMISSED

- 显示“已撤回”。
- 只读，不继续占用主要操作层级。

### APPLIED

- 显示“已写入项目”。
- 展示最多三个实际路径，更多内容显示“另有 N 个文件”。
- 提供“查看文件”和“查看 Git 变更”。
- 不允许重复应用。

### PARTIALLY_APPLIED

- 显示“部分文件已写入”。
- 明确分组展示成功和失败结果。
- 提供“查看文件”“查看 Git 变更”和“仅重试失败项”。
- 不允许重新应用成功项。

所有操作按钮使用图标加文字，最小触控高度为 `48dp`。只有真实错误使用错误红色，成功和普通操作沿用当前珊瑚色主题。

## 系统事件

所有文件成功时，持久化系统事件记录实际写入路径。

部分成功时，系统事件同时记录：

- 成功写入数量和路径。
- 失败数量和路径。
- 不包含敏感文件正文的简短失败原因。

系统事件是会话历史记录；带跳转动作的结果卡片只保证当前聊天页面生命周期内可用。

## 错误处理

- 无项目：提示“请先选择项目”，不创建草稿。
- 空输入：不发送。
- 图片输入：提示“文件变更模式暂不支持图片输入”。
- 助手忙碌：不重复发送。
- LLM 非结构化结果：进入 `FAILED`，保留原始请求供重试。
- 空更新计划：进入 `FAILED`。
- 全部撤回：进入 `DISMISSED`，不写文件。
- 单文件失败：记录失败项并继续处理其他项。
- 全部写入失败：进入 `FAILED`，保留 diff 和失败项。
- 部分写入失败：进入 `PARTIALLY_APPLIED`，保留成功文件并只重试失败项。
- 项目已删除：项目页进入空状态并提示项目不存在。
- Git 状态读取失败：不影响已写入文件，项目页展示可重试错误。

## 测试策略

### 单元测试

`MarkdownFileChangeControllerTest`：

- 覆盖全部六种状态。
- 覆盖成功路径和失败项记录。
- 覆盖部分成功后的失败项重试。
- 验证重试不包含已成功项。

Gateway 和 Repository 测试：

- 模拟第二个文件失败，验证第一个文件真实存在。
- 验证失败后仍继续处理后续文件。
- 验证批量结果正确区分成功和失败。
- 覆盖路径逃逸、非 Markdown 路径、空路径和项目不存在。

导航状态测试：

- “查看文件”生成正确的项目、标签页和路径目标。
- “查看 Git 变更”生成正确的项目和 Git 目标。
- 项目页只消费一次请求。
- 文件目标触发文件刷新，Git 目标触发 Git 刷新。

### Compose UI 测试

- `APPLIED` 卡片展示路径和两个查看操作。
- `PARTIALLY_APPLIED` 展示成功与失败分组。
- “仅重试失败项”可点击，成功项没有重复应用入口。
- 浅色、深色和大字体下状态文字与按钮不重叠。

### 模拟器与真实设备验收

1. 创建项目并新建项目会话。
2. 生成至少两个 Markdown 变更。
3. 审核并应用。
4. 验证聊天页展示真实写入结果。
5. 点击“查看文件”，确认文件夹刷新并选中首个文件。
6. 返回聊天，点击“查看 Git 变更”，确认工作区变更可见。
7. 制造一个单文件写入失败，确认其他文件保留。
8. 仅重试失败项，确认成功项没有被重复执行。

## 完成标准

- 项目会话能明确区分普通发送和文件变更。
- 所有 LLM 文件写入都经过 diff 审核。
- 成功写入结果在聊天、项目文件夹和 Git 状态中一致可见。
- 部分失败不会丢失或隐藏成功结果。
- 失败项可以独立重试。
- 查看文件和查看 Git 变更不会自动提交或推送。
- 自动化测试、Debug 构建、设备测试和人工主链路验收通过。

满足以上标准后，第一阶段结束，再进入“项目工作台闭环”第二阶段设计与实施。
