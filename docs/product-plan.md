# Harness APK 产品计划

## 产品定位

Harness APK 是一个移动端个人 AI 项目工作台。它面向需要在手机上持续推进项目的人，提供 LLM 会话、项目资料沉淀、Markdown 文件变更审核、Git 状态查看、模型能力配置和 APK 自更新能力。

产品不定位为通用聊天应用，也不优先追求社交、账号、云端协同或应用商店分发。当前阶段的核心目标是让一个可信用户在 Android 手机上完成“提问、生成、沉淀、审核、维护项目资料”的闭环。

## 当前状态

### 已具备能力

- 多会话聊天、本地会话和消息保存。
- Provider 配置、本地加密保存 API Key、模型列表和默认模型维护。
- 文本流式回复、图片附件输入和图片压缩。
- 长上下文自动压缩、模型上下文窗口配置和压缩阈值配置。
- 联网搜索配置，支持外置搜索和部分 Provider 原生搜索模式。
- 会话和项目双模式首页。
- 项目空间、项目会话、项目文件夹和多类型项目文件索引。
- Markdown 文件变更计划、diff 审核、应用写回和项目文件预览。
- PDF 导出、项目 ZIP 导入导出。
- Git / Gitee 认证配置、提交身份配置和项目 Git 面板基础能力。
- 技能开关和内置技能展示。
- APK 更新检查、下载、校验和安装入口。

### 已形成设计方向

- 项目是长期工作的容器，会话是项目工作的主入口，文件夹是沉淀产物视图。
- Markdown 生成和修改应以文件变更卡片呈现，完整正文进入 diff 审核和项目文件，不把长文档塞进聊天气泡。
- 普通会话保持轻量续聊效率，项目会话额外承载项目上下文和写回能力。
- Provider 能力应从简单开关逐步走向能力矩阵，包括文本、视觉、搜索、上下文窗口和模型参数。

### 主要缺口

- 项目工作台闭环还需要更稳定地连接“生成文件变更、审核、写回、查看 Git 状态、提交”。
- 文件变更体验需要从 Markdown 扩展到更多项目产物的可预览、可导出和可追踪。
- Git 能力仍偏设置和状态，缺少面向移动端的低风险提交、差异确认和推送流程。
- 模型能力配置需要降低手工维护成本，减少用户因模型能力不匹配导致的失败。
- 自更新、发布通道和回归验证需要形成固定节奏。

## 阶段目标

### P0：稳定可用闭环

目标是在真实 Android 设备上稳定完成一次完整项目工作流：

1. 创建或导入项目。
2. 新建项目会话。
3. 使用 LLM 生成或更新 Markdown。
4. 审核 diff 并写回项目文件。
5. 在项目文件夹查看结果。
6. 查看 Git 状态。
7. 通过自更新获得新版本。

P0 不追求多用户协作、云同步和复杂插件执行，优先消除核心闭环里的阻塞、误写和性能问题。

### P1：项目工作台增强

目标是让项目页成为手机端长期工作的默认入口。

重点包括项目会话管理、文件夹浏览、文件搜索、项目上下文维护、Markdown/PDF 预览、项目导入导出和 Git 变更理解。P1 应避免把项目页做成复杂 IDE，移动端只保留高频、低风险、可确认的操作。

### P2：模型与工具能力扩展

目标是让不同 Provider、模型和工具能力可被稳定选择、解释和降级。

重点包括模型能力矩阵、上下文窗口配置、搜索能力路由、视觉模型选择、语音输入和技能开关。P2 的原则是能力透明：用户应能理解当前会话用了哪个 Provider、哪个模型、是否启用搜索、是否支持图片，以及失败时应该如何调整。

### P3：分发、更新和长期维护

目标是让测试包和正式包可以持续发布，出现问题时可以快速定位和回退。

重点包括测试/正式通道分离、发布清单、SHA-256 校验、更新红点和静默下载策略、发布前回归清单、关键错误诊断和敏感信息脱敏。

## 研发路线图

### 近期：P0 稳定可用闭环

时间粒度：1-2 个迭代。

范围：

- 梳理项目会话到文件变更卡片的主路径，减少“普通回复”和“文件变更”之间的混淆。
- 完善 Markdown 变更卡片的失败、重试、撤回和应用成功状态。
- 确保写回项目后的文件夹、预览和 Git 状态能及时刷新。
- 补齐项目导入、导出、PDF 导出和路径安全相关回归测试。
- 固化真实设备验收流程，覆盖 Provider 配置、会话发送、项目写回和 APK 更新。

验收标准：

- 用户无需理解文件系统细节，即可从项目会话生成并写回一份 Markdown。
- 任意写文件动作都必须经过可见审核，不允许静默覆盖项目文件。
- 长 Markdown 不会拖垮聊天流式渲染。
- API Key、图片内容和完整 prompt 不出现在日志、错误提示或导出配置中。
- 测试通道 APK 可以被检查、下载、校验并拉起系统安装器。

### 中期：P1 项目工作台增强

时间粒度：2-4 个迭代。

范围：

- 优化项目首页的信息架构：项目切换、项目会话、文件夹、Git 状态各自边界清晰。
- 文件夹支持更好的筛选、搜索、预览和外部打开。
- 项目上下文成为会话请求的稳定输入，并支持用户明确维护。
- Git 面板支持移动端友好的 diff 摘要、提交前确认和低风险提交动作。
- 项目 ZIP 导入导出补齐错误提示、冲突处理和大文件限制。

验收标准：

- 用户可以把一个项目作为长期容器持续使用，而不是每次从普通会话重新开始。
- 项目文件、项目上下文和项目会话之间的关系可解释、可恢复、可测试。
- Git 操作不会绕过用户确认，不会默认提交无关文件。
- 项目导入失败时能说明具体原因，不留下半导入状态。

### 中后期：P2 模型与工具能力扩展

时间粒度：2-3 个迭代。

范围：

- 建立 Provider 能力清单：文本、视觉、搜索、上下文窗口、推理参数和模型别名。
- 让会话页显示当前模型能力摘要，发送前识别图片、搜索、长上下文等能力不匹配。
- 优化搜索能力：区分外置搜索、模型原生搜索和禁用搜索。
- 完善语音输入和朗读，先服务移动端快速输入，不做复杂语音助手。
- 技能能力继续保持默认关闭，开放前补齐来源、权限、只读和执行边界说明。

验收标准：

- 用户能在发送前知道当前模型是否支持本次输入。
- 模型能力不匹配时给出可执行建议，而不是只显示请求失败。
- 搜索来源能在回复中可见，失败时可降级到普通聊天。
- 技能和插件不会在用户不理解权限边界时自动执行。

### 长期：P3 分发与维护体系

时间粒度：持续维护。

范围：

- 建立每次发布前的固定回归清单。
- 测试通道优先自动部署，正式通道保留人工确认。
- 更新 manifest 保持向后兼容，支持强制更新和普通更新。
- 关键异常保留诊断信息，但持续检查敏感信息脱敏。
- 按版本维护已知问题、修复记录和回滚说明。

验收标准：

- 测试包可以稳定分发给目标设备。
- 正式包不会因签名、manifest 或 SHA-256 配置错误导致不可安装。
- 用户遇到更新失败时能保留当前可用版本，并知道下一步操作。
- 研发侧能从脱敏诊断日志定位 Provider、模型、网络、更新或文件系统问题。

## 优先级原则

1. 安全优先：API Key、项目文件、图片和 prompt 不应被误写、误传、误导出。
2. 闭环优先：优先完成会话到项目文件再到 Git 状态的闭环，而不是横向增加入口。
3. 审核优先：任何项目文件写入、Git 提交和发布动作都需要用户确认。
4. 稳定优先：长文本、流式渲染、上下文压缩和更新下载要优先避免卡顿和不可恢复失败。
5. 移动端优先：手机上只做高频、明确、低风险操作，复杂批量编辑和高风险 Git 操作不作为近期目标。

## 近期执行清单

- 明确文件变更模式的主入口和普通发送入口，减少用户误用。
- 完善 Markdown 文件变更卡片状态机：生成中、已生成、失败、已撤回、已应用。
- 写回项目后刷新文件夹、文件预览和 Git 状态。
- 给项目会话增加更清晰的当前项目和当前模型提示。
- 补齐项目上下文进入 LLM 请求的展示和测试。
- 补齐 Git 面板的移动端 diff 摘要和提交前确认方案。
- 强化 Provider 能力校验：图片、搜索、上下文窗口和模型参数。
- 整理测试/正式通道发布检查清单。
- 为 APK 更新失败、校验失败和取消安装补齐用户可理解提示。
- 扩展回归测试，覆盖项目写回、长 Markdown 渲染、Provider 能力和更新流程。

## 暂不做

- 用户账号、家庭成员体系和云端会话同步。
- 应用商店发布、支付、订阅和额度管理。
- 静默安装、后台无感升级和高风险系统权限。
- 自动提交、自动推送或无审核写入项目文件。
- 复杂插件执行平台和第三方代码运行。
- 完整 IDE 式代码编辑器。
